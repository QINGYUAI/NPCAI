## 0. 摘要（TL;DR）

M4.2 交付了 5 个独立子系统（预算 / 观测 / 记忆 / 反思 / 事件），但**缺少横向串线**：

1. 一次 tick 内的 LLM 调用、记忆写入、反思产物、事件生成之间**无统一 id**，线上问题回溯要手拼 `scene_id + npc_id + 时间窗`；
2. NPC 之间**无直接交互路径**：`neighbors` 只提供名字列表，NPC A 说的话不会进入 NPC B 的下一 tick 输入。

M4.3 用**两步纵切**解决这两条主干：

| 子节点 | 目标 | 预估 | 依赖 |
|---|---|---|---|
| **M4.3.0** | tick 粒度 `trace_id` uuid 贯穿 ai_call_log / npc_tick_log / scene_event / npc_memory / npc_reflection | 0.5 天 | — |
| **M4.3.1.a** | `scene_event` 扩列（`parent_event_id / conv_turn`）+ plan 结果自动写 dialogue event | 0.75 天 | M4.3.0 |
| **M4.3.1.b** | Engine Node：NPC B 在下一 tick 感知到 NPC A 的 dialogue event，带硬上限回声保护 | 1 天 | M4.3.1.a |
| **M4.3.1.c** | 前端：📢 事件抽屉高亮 dialogue 分组 + 状态气泡展示"谁在跟谁说话" | 0.5 天 | M4.3.1.b |

**总工作量 ≈ 2.75 天**（单人；含联调与单测；不含 live smoke buffer）。

**架构原则延续 M4.2**：每个子批次独立可回滚；不引入 LangGraph.js（继续保留 M4.4 重构窗口）；全节点节点内降级，不阻主 scheduler 循环。

## 1. 范围

### 1.1 范围内（M4.3）

1. **M4.3.0 trace 贯穿**：新增 tick 级 `trace_id`（uuid v4），在 5 张表 + WS 事件中打穿；新增运维探针按 `trace_id` 归集整条 tick 链路。
2. **M4.3.1 多 agent 交互链**：NPC A 的 `plan.say` 产物**自动注入**为 `scene_event{type='dialogue'}`，次 tick 被同场景 NPC 作为 `eventBlock` 输入；硬上限回声保护防循环；前端可视化复用现有组件。

### 1.2 非范围（M4.4+ 再考虑，与 M4.2 §1.2 排除项并列）

**已评审确认本节点不做**：

- **日程 / 长期规划**（M4.2 summary §7.2）：缺虚拟游戏时间层作前置，推迟至 M4.4。
- **Sandbox.vue 拆分**（M4.2 summary §7.4）：67.7 KB 偏胖是技术债，但本节点不动 UI 骨架；留到 M4.4 前端重构窗口统一做。
- **观测性 P95 / Prometheus**（M4.2 summary §7.5）：trace_id 是它的前置，但埋点 + 时序面板本身不进 M4.3。
- **LangGraph.js 接管推理图**（M4.2 §1.2 延续）。
- **Human-in-the-loop**：玩家插话 / 打断对话，留 M4.4。
- **NPC 动作真实副作用**：`plan.action` 仍不改坐标/库存/关系图（延续 M4.2）。
- **turn-taking 调度**：不引入回合制/发言权管理，保持 scheduler 并行 + 事件异步传播模型。
- **对话 session / conversation 实体**：Q3 已明确**不建 `conversation` 表**；同一对话的聚合靠 `parent_event_id` 链表回看，不做范式化。

### 2.2 新增环境变量（backend/.env）

```env
# M4.3.0 trace 贯穿
TRACE_ID_ENABLED=true              # false 时跳过生成 + 入库（字段允许 NULL），方便临时回滚
TRACE_ID_LOG_FORMAT=short          # short=首 8 字节缩写打印；full=完整 uuid（排障用）

# M4.3.1 多 agent 对话自动化
DIALOGUE_AUTO_EVENT_ENABLED=true   # false 时完全跳过 plan→dialogue event 注入（回到 M4.2 行为）
DIALOGUE_ECHO_WINDOW_TICK=10       # 回声保护窗口（近 N tick 内）
DIALOGUE_ECHO_MAX_TURN=3           # 同一 NPC pair 在窗口内最多互相回应的轮数
DIALOGUE_CONTENT_MAX_LEN=200       # plan.say 入事件 content 的截断长度（超出加 …）
```

**env 默认值策略**：全部开关默认 `true`，保持"升级一把刀"的风格；下线任一子系统只需单独 flip env，无需回滚代码。

## 3. 数据层变更

### 3.1 现有表追加列（**无新建表**，Q8 锁定）

| 表 | 列 | 默认 | 用途 | 子里程碑 |
|---|---|---|---|---|
| `npc_tick_log` | `trace_id CHAR(36) DEFAULT NULL` | NULL | tick 级 uuid，与本 tick 下所有其他表一致 | M4.3.0 |
| `ai_call_log` | `trace_id CHAR(36) DEFAULT NULL` | NULL | 同 tick 内一次/多次 LLM 调用归集 | M4.3.0 |
| `scene_event` | `trace_id CHAR(36) DEFAULT NULL` | NULL | 事件产生时所在的 tick（被动注入事件写 API-tick 当前值） | M4.3.0 |
| `scene_event` | `parent_event_id BIGINT DEFAULT NULL` | NULL | dialogue 回复链条：若 A 的 say 是针对 B 早先的 say，则填 B 的 event_id；NULL=会话起点 | M4.3.1.a |
| `scene_event` | `conv_turn INT DEFAULT NULL` | NULL | 本次对话链上的轮序（起点=1，每回复 +1）；非 dialogue 事件为 NULL | M4.3.1.a |
| `npc_memory` | `trace_id CHAR(36) DEFAULT NULL` | NULL | memory-store 写入时复制当时 tick trace_id | M4.3.0 |
| `npc_reflection` | `trace_id CHAR(36) DEFAULT NULL` | NULL | 反思触发的那 tick trace_id；注意与 `source_memory_ids` 里的 trace 可能不同 | M4.3.0 |

**索引策略**（M4.3.0）：

```sql
CREATE INDEX idx_npc_tick_log_trace   ON npc_tick_log(trace_id);
CREATE INDEX idx_ai_call_log_trace    ON ai_call_log(trace_id);
CREATE INDEX idx_scene_event_trace    ON scene_event(trace_id);
CREATE INDEX idx_npc_memory_trace     ON npc_memory(trace_id);
CREATE INDEX idx_npc_reflection_trace ON npc_reflection(trace_id);
```

每表单列二级索引；前缀 36 字节 B-tree 足够选择（单 tick 全局唯一），不必复合索引。

**dialogue 链索引**（M4.3.1.a）：

```sql
CREATE INDEX idx_scene_event_parent ON scene_event(parent_event_id);
```

用于「打开一条 say 查它是回复谁」的回溯查询。

## 5. 引擎内部变更

### 5.1 推理图扩展（graph/build.ts）

**M4.2 已有**：`event-intake → memory-retrieve → plan → speak → memory-store → memory-summary → reflect?`

**M4.3 增量**（全部在原节点内部，**不加新节点**）：

1. `plan` 节点输出后：若 `plan.say` 非空 **且** `DIALOGUE_AUTO_EVENT_ENABLED=true`：
   - 调用新增 `engine/dialogue/emit.ts::emitDialogueFromSay(sceneId, npcId, plan.say, traceId, parentEventId?)`
   - 内部走 `POST scene_event` 相同入库路径（**复用 M4.2.4 的 insert 函数**而非 controller），保证 WS 同步 emit；
   - 同时写 `parent_event_id + conv_turn`：若本 NPC 当前 tick 的 `eventBlock` 里**命中过一条 dialogue event**，则取其中 `conv_turn` 最大且 actor≠self 的那条作为 parent；否则 parent=NULL / conv_turn=1。
2. `event-intake` 节点：`pickEventsForNpc` 纯函数新增**回声保护过滤**：
   - 若 (NPC_A, NPC_B) 在最近 `DIALOGUE_ECHO_WINDOW_TICK` 个 tick 内已互相 dialogue `DIALOGUE_ECHO_MAX_TURN` 轮，则本 NPC 的 eventBlock 丢弃这条 dialogue event（进 `dropped_count`，并 `console.warn`）。
   - 判定用 `scene_event` 的 `(actor, parent_event_id 链)` 回溯，不依赖新表。
3. `scheduler.ts::runTick` 开头：生成一次 `traceId = crypto.randomUUID()`，存进 `TickContext`，本 tick 内所有数据库写入（含 memory-store / reflect / dialogue emit）从 `TickContext` 读同一 id。

### 5.2 dialogue 自动化的"隐式"契约（Q2 锁定）

**不做语义判断**：不用新的 LLM 调用判断"NPC A 在对 B 说话"；规则是：

- `plan.say` 非空 → 必产一条 `scene_event{type:'dialogue', actor:<npc_name>, content:<say 截断 200 字>, visible_npcs:null}`；
- 谁"听到"由事件可见性（Q5：同场景全 NPC）决定，不做发言对象路由；
- 次 tick `event-intake` 时，**除说话人自己以外**的同场 NPC 看到这条事件，可能（由 LLM 自主决定）在他们自己的 `plan.say` 里回应；
- 若回应，`emitDialogueFromSay` 把 parent 指回被回应的 event_id（取当前 `eventBlock` 里 actor≠self 的最新 dialogue event），自然形成链表。

**链式结构**示意（LeLei ↔ HanMeimei）：

```
event 33  actor=李雷   content="吃了吗？"           parent=null conv_turn=1 trace=T1
event 41  actor=韩梅梅 content="吃过了你呢？"       parent=33   conv_turn=2 trace=T2
event 49  actor=李雷   content="我也刚吃完。"       parent=41   conv_turn=3 trace=T3
event 57  actor=韩梅梅 content="那晚上一起散步？"   parent=49   conv_turn=4 trace=T4  ← 达到 N=3 (李雷↔梅梅 3 轮) 后下一轮开始被回声保护丢弃
```

### 5.3 scheduler.ts（M4.3.0）

- 在 `bus.emitEvent({type:'tick.start', ...})` 之前生成 `traceId`，并附加到 event payload；
- `TickContext`（内存结构）新增 `traceId: string` 字段，贯穿本 tick 所有节点；
- `tick.npc.updated` / `tick.end` / `meta.warn` / `reflection.created` / `scene.event.created` 的 WS 序列化都附带 `trace_id`；
- `persistNpcTick` 的 INSERT 参数从 `TickContext` 拿 trace_id；
- env `TRACE_ID_ENABLED=false` 时 `traceId=null`，所有下游字段写 NULL，无副作用。

### 5.4 降级策略

- `emitDialogueFromSay` 失败 → `console.warn`，**不阻** plan→speak→memory-store 主链路；
- 回声保护判定 SQL 失败 → 放行（宁可多一条不阻止），记 warn；
- `crypto.randomUUID` 不会失败（Node 内置），但 `TRACE_ID_ENABLED=false` 走 null 分支一次性跳过。

## 8. 关键设计决策（Q1~Q9 拉票定稿）

| 编号 | 问题 | 选项 | **定稿（方案甲）** | 理由 |
|---|---|---|---|---|
| Q1 | 本节点范围 | a: M4.3.0 + M4.3.1 四批 / b: 只做 M4.3.0 / c: 只做 M4.3.1 / d: 全做 + 观测面板 | **a** | b 无消费者价值低；c 缺 trace 回溯断链；d 超 3 天打破节奏 |
| Q2 | plan→dialogue 转换 | a: 只要 `plan.say` 非空就转 / b: 再加 LLM 语义判断是否在对谁说话 / c: 关键词启发式 | **a** | b 加 1 次 LLM 翻倍成本且 zod 难收敛；c 中文易误判；a 简洁 + 靠事件可见性自调节 |
| Q3 | 会话建模 | a: 无 conversation 表，每 say = 独立 event，parent_event_id 链回看 / b: 建 `conversation` 聚合表 | **a** | 减 1 张表 1 次迁移；跨场景 / 跨对聚合靠查询层的 CTE 即可；建表会绑死"对话"语义边界 |
| Q4 | 回声保护 | a: 硬上限 `window=10, N=3` / b: 动态指数退避 / c: 完全放开靠 LLM 自控 | **a** | b 调参成本高；c 会无限循环烧 token；a 可观测可回退 |
| Q5 | dialogue 可见性 | a: `visible_npcs=null` 同场全可见 / b: 只对 @mention 的对象 / c: 发起者选择 | **a** | b 需 LLM 识别@语义；c 需 plan schema 扩字段；a 顺承 M4.2.4 默认行为 |
| Q6 | trace_id 粒度 | a: tick 级 / b: scene 级 / c: NPC 级 / d: API 请求级 | **a** | scene 级太粗（跨 tick 无法区分）；NPC 级太细（一次 tick 要串 N 个 trace）；tick 是 runGraph 的自然边界 |

## 9. 风险与降级

| 风险 | 级别 | 降级方式 |
|---|---|---|
| dialogue 自动化导致 tick 内新增 1 次 DB 写 → QPS 涨 N × tick | 中 | `DIALOGUE_AUTO_EVENT_ENABLED=false` 一键关；短期内每 NPC 每 tick 至多 +1 INSERT，量级仍远低于 memory-store（至少 +2） |
| 回声保护 bug 导致对话进入死循环烧 token | 高 | 硬上限双保险：① N=3 在 intake 层拦截；② 预算硬检查（M4.2.0）在 scheduler 层兜底；③ 失败时 env 关闭自动化回到 M4.2 行为 |
| `crypto.randomUUID` 在 Node < 19 环境不存在 | 低 | 启动时 `assert(typeof crypto.randomUUID === 'function')`；README 声明最低 Node 19（当前项目已是 Node 20+） |
| 前端气泡回应后缀 flicker | 低 | 气泡层本就是 3s 轮询，回应后缀仅靠 `latest_say` 映射到最新 dialogue event 做查找；延迟 ≤3s 可接受 |
| 跨 tick 的 trace_id 泄漏（NPC A trace 写到 NPC B 行） | 中 | 单测强断言：`runTick` 开头生成后只经 `TickContext` 传递，不放全局变量；review checklist 新增一条"trace_id 只从 context 读" |
