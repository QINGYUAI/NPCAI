## 0. TL;DR

- **前因**：M4.3 关盘后确认 3 项核心能力缺口（见 `m4.3-summary.md §7`）：
  1. **对话链被拆断**（L-1 / L-4）：`EVENT_LOOKBACK_SECONDS=60s` + budget skip 会把跨 tick 对话链拉断；`DIALOGUE_ECHO_WINDOW_TICK` 读而不用
  2. **NPC 缺长期驱动力**：只对当前事件/记忆反应，无日程 / 无目标 / 无时间感
  3. **前端气泡空白态**：闲时 NPC 无 `latest_say/latest_action` 时气泡空，沙盒视觉体验差
- **交付**（roadmap 方案甲 Q1-Q9 全 a 定稿）：
  - **M4.4.0 · T-A 对话链窗口治理**：`scene_event.created_tick` 列 + 混合窗口（时间 OR 条数取并集）+ `DIALOGUE_ECHO_WINDOW_TICK` 真正按 tick 差判拦 → 彻底解 L-1 / L-4
  - **M4.4.1.a · T-B 数据层 + schedule 模块**：`npc_schedule` 表（24 hour × N NPC）+ `engine/schedule/{config,resolve,fetch}.ts` 三件套 + scheduler 注入
  - **M4.4.1.b · T-B plan 节点 + 前端气泡**：plan prompt 日程驱动分支（事件非空→忽略、事件空+有日程→注入）+ 前端气泡「📅 当前日程」闲时回退
- **规模**：后端单测 180 → **205**（+25，超 roadmap 目标 +22）；前端单测 15 → **18**（+3，正好命中）；新表 1 / 扩列 1 / 新 env 6 / 新 REST 0 / 新 WS 事件 0 / 新前端组件 0（继续 Q7=a 最小改动）
- **工时**：roadmap 估 3 天；实际 ~1 天（M4.4.0 在上游 session 预先完成，本 session 负责 M4.4.1.a/b 全实现 + 整节点关盘）

## 2. 数据层改动（M4.4 合计）

### 2.1 新表（+1）

```sql
CREATE TABLE npc_schedule (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  npc_id      BIGINT NOT NULL,
  hour        TINYINT NOT NULL COMMENT '0..23 小时槽',
  activity    VARCHAR(128) NOT NULL,
  location    VARCHAR(64) DEFAULT NULL,
  priority    TINYINT DEFAULT 5 COMMENT '1..10',
  updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_npc_hour (npc_id, hour),
  INDEX idx_npc_schedule_npc (npc_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.2 扩列（+1）

| 表 | 列 | 类型 | 说明 |
|---|---|---|---|
| scene_event | `created_tick` | INT NULL | 事件产生时的 tick 号，给 DIALOGUE_ECHO_WINDOW_TICK 精确判窗口（解 L-4） |

### 2.3 新索引（+2）

- `scene_event.idx_scene_event_created_tick`
- `npc_schedule.idx_npc_schedule_npc` + `uk_npc_hour` 唯一键

### 2.4 迁移脚本（+1）

- `backend/src/db/migrate-m44.ts` 一个文件分 2 段幂等执行：
  - 段 1（M4.4.0）：`scene_event.created_tick` 列 + 索引（列存在跳过 / 索引存在跳过）
  - 段 2（M4.4.1.a）：`npc_schedule` 建表 + seed 24h × N NPC UPSERT（表存在跳过建表 / seed 用 `ON DUPLICATE KEY UPDATE`）

## 3. 环境变量（+6）

```env
# M4.4.0 对话链窗口治理
EVENT_LOOKBACK_SECONDS=120              # 原 60 → 120（Q2=a 时间兜底）
EVENT_LOOKBACK_COUNT=50                 # 新增：条数窗上限
EVENT_MAX_BLOCK_ITEMS=30                # 新增：intake prompt 去老化上限
# DIALOGUE_ECHO_WINDOW_TICK=10          # 原存在，M4.4.0 起真正消费

# M4.4.1 NPC 日程
SCHEDULE_ENABLED=true                   # 总开关；false 时回 M4.4.0 行为
SCHEDULE_LLM_HINT=false                 # Q4=b 预留通道（并行 hint）本批不消费
# SIM_CLOCK_HOUR=<0..23>                # 可选：单测/演示时注入固定 hour
```

**回退通道**：
- `EVENT_LOOKBACK_COUNT=0` → 纯时间窗，等价 M4.3 行为
- `DIALOGUE_ECHO_WINDOW_TICK=0` → 禁用 tick 差判拦，echo 回退 M4.3 隐式窗口
- `SCHEDULE_ENABLED=false` → scheduler 不查 `npc_schedule`，`scheduledActivity=null`，plan/bubble 回 M4.4.0 行为

## 4. 观测与运维

### 4.1 新增 REST

本节点**零新增 REST**。M4.3 的 `GET /api/engine/trace/:trace_id` 可对账日程/事件同 tick 写入（会读到 `scene_event.created_tick`）。

### 4.2 结构化日志

- `[schedule.fetch] 查询失败，降级为空日程` — DB 异常或表缺列时的降级信号
- 之前的 `[intake.echo] 回声拦截 ...` 日志仍在；tick 差窗口命中时拦截数可能更低（更"宽松"）

### 4.3 WS 事件

- 继承 M4.3：`scene.event.created` 携带 `created_tick?`（前端可选用，当前只在气泡气泡状态里间接体现）

## 7. 已知限制 & 延后项

| # | 描述 | 级别 | 落地节点 |
|---|---|---|---|
| N-1 | `scheduled_activity` 仅用于 plan 节点 system prompt + 前端气泡回退；memory/reflection 侧未纳入（不会影响长期记忆的"时间感"） | 低 | M4.5：可考虑让 memory-summary 把日程当一个 slot 写入 |
| N-2 | 日程模板是 DB seed 的固定 24 小时；无"动态目标"/"随机打断"机制（如"今天 10 点临时有访客"） | 中 | M4.5 或 M5.x：可接 scene 事件 → 临时覆盖日程 |
| N-3 | `SCHEDULE_LLM_HINT=true` 通道预留但未实现（Q4=b 方案，即并行注入到 user 消息末尾作为 hint） | 低 | 按需启用；现阶段前置分支足够 |
| N-4 | `npc_schedule.priority` 列存在但 plan 节点未按 priority 做"当前时段 vs 下一时段"的平滑过渡（整点切换可能带来对话风格突变） | 低 | M4.5：加 15 分钟 soft window 插值 |
| N-5 | 前端气泡只展示 `activity + location`，不展示 `priority`/下一时段提示 | 低 | M4.5：可选 tooltip |
| N-6 | 整节点 live smoke 通过"手插事件 + pure-function 验证"等价验 4 个子功能，未做 4 NPC × 20 tick 大规模跑（同 M4.3 L-6） | 低 | 未来 scale-out 时统一跑 |

**明确延后**：LangGraph.js 迁移（Q5=a）/ 动态日程覆盖（N-2）/ LLM hint 并行通道（N-3）/ soft window 平滑（N-4）—— 全部归入 M4.5 窗口。
