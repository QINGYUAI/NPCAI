## 4. 核心技术决策（贯穿 M4.2）

| 决策点 | 选择 | 理由 |
|---|---|---|
| Tokenizer | `tiktoken`（wasm） | provider 不返 usage 时兜底；硬编码单价表避开 API 漂移 |
| 向量库 | **Qdrant**（放弃 MySQL 客户端余弦） | 初版就上成熟库避免自维护 ANN |
| WebSocket | `ws` + express 同端口 | 不另起进程；端口/网络面一致 |
| Embed model 切换 | pointer 式（`ai_config.id`）| 60s TTL 缓存；chat/embed provider 分离但凭据单一入口 |
| 反思主题 | 固定 3 主题（goal/emotion/relation） | 与 `simulation_meta.plan` 锁 schema；完备性校验可机械化 |
| 事件去重 | per-(event_id × npc_id) 独立表 | 比 in-memory Set 精确；FK CASCADE 维护成本低 |
| 事件查询 | tick 顶部**一次** DB 查询 + 内存分发 | N² SQL → 1 SQL；scheduler 热路径只读 |
| 降级策略 | **全节点节点内降级，不阻主流程** | event-intake 失败 → empty；embedding 失败 → pending 状态保留；Qdrant 失败 → MySQL importance 排序 |
| 前端通信 | WS 优先 + 连续失败 5 次 → 3s 轮询 | 指数退避 1/2/4/8/16s；空闲 45s 主动重连 |
| Ring Buffer 长度 | 统一 20 条（timeline/reflection/event） | 场景首屏同步补发 + 内存 footprint 可控 |

## 7. M4.3 建议议题（供下一次评审）

1. **多 agent 交互链**：NPC 之间直接触达 event / say，不走 scheduler 轮询（借 M4.2.4 事件总线做桥梁）
2. **日程 / 长期规划**：结合记忆 + 反思产出日/周级 plan，scheduler 加载为 precondition
3. **trace id 贯穿**（X-8）：tick 粒度 uuid 注入全部子系统日志
4. **前端 NPC 卡片拆分**：当前 `Sandbox.vue` 67 KB 已偏胖，建议按「左布局 / 右面板 / 抽屉族」拆 3 层
5. **观测性 P95**（X-1）：tick 延迟、LLM 延迟、WS 连通率 3 指标接 Prometheus / 时序面板
