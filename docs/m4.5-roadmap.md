## 0. 摘要（TL;DR）

M4.4 交付后，NPC 有**日程骨架 + 对话可续接**，但仍缺 3 类能力：

1. **动态主动性** —— 日程是"硬编码 24 小时"，没有**临时目标**/**剧情插入**/**玩家干预**的覆盖机制（N-2）；NPC 的记忆里也没有"时间感"，反思写的都是"发生了什么"而非"我在什么时段做了什么"（N-1）
2. **工程可维护性** —— `Sandbox.vue` 从 M4.2 开始挂账至今已**延后 3 个节点**，1800+ 行单文件（X-3）；P95 observability 自 M4.2 起未落地（X-1）；反思周期仍然是全局一刀切（X-5）
3. **架构跃迁窗口** —— LangGraph.js 连续 2 节点挂账（M4.4 Q5=a）；现有 `graph/build.ts` 手写编排能跑但扩展性受限，尤其 M4.5 引入动态目标后，plan 节点分支会从「event vs schedule」升级到「event vs goal vs schedule」三路（Q5=a 是否还能顶？）

本节点目标：**从 3 类能力里挑 2 条落地，保持 2~3 天预算**。

## 2. 组合方案（4 套）

### 方案甲 · 主动性加深（推荐，2.5~3 天）

```
U-A（动态目标系统）→ U-B（记忆带时间感）→ U-N（日程 soft window）
```

- **逻辑**：承接 M4.4 的日程骨架，补齐"目标覆盖 + 记忆时间感 + 整点平滑"三件套，NPC 从"有日程但机械"升级到"会被剧情/玩家动态影响"
- **收益**：**演示效果再跃迁**——玩家可通过 REST 给 NPC 下临时目标（如"去找小美聊聊"），NPC 当下放弃日程响应；反思里也能体现"下午我在书房读书时小美来找过我"
- **风险**：U-A 新表 + 新接口 + plan 节点前置分支从 2 路变 3 路，回归面中等；U-B 改 reflect 可能影响现有 summary 质量，需分批 live smoke
- **预估**：1.8 + 0.6 + 0.5 = **2.9 天**（含 live smoke）

### 方案乙 · 工程债务清理（1.5~2 天）

```
U-C（Sandbox.vue 拆分）+ U-D（观测 P95）+ U-E（反思差异化）
```

- **逻辑**：集中清 M4.2 三项核心延后（X-3/X-1/X-5），**0 新功能、0 用户可见变化**
- **收益**：后续迭代速度显著提升；观测接入后可客观评估成本优化效果
- **风险**：纯工程不刺激，但 X-3 已延后 3 节点，再不做容易积成技术债翻车
- **预估**：1 + 0.8 + 0.4 = **2.2 天**

### 方案丙 · 架构跃迁（3 天，高风险）

```
U-F（LangGraph.js 迁移）+ U-L/U-O（轻量搭配）
```

- **逻辑**：趁 M4.5 没有巨大新功能压力，一次性把图引擎换掉，**为 M4.6+ 多 agent / 并行 / 分支留空间**
- **收益**：架构债务一次清掉；condition edges 让 plan 分支从命令式变声明式
- **风险**：LangGraph.js 学习曲线 + 全量回归（现有 152+ 单测要逐条对齐）；**若回归不完整，M4.6 会持续吃成本**
- **预估**：2.5 + 0.3 + 0.3 = **3.1 天**（**无 buffer**）

### 方案丁 · 主动性 + 工程混合（2.5 天）

```
U-A（动态目标）+ U-C（Sandbox 拆分）
```

- **逻辑**：主动性只做最核心的 U-A，工程只动最紧的 U-C
- **收益**：NPC 主动性 + 前端可维护性双线推进
- **风险**：U-A 的 reflect/memory 没升级，NPC 还是"对目标机械执行"；Sandbox 拆分后可能需要 0.3 天补 bug
- **预估**：1.8 + 1 = **2.8 天**

## 3. 推荐：方案甲（2.9 天）

**理由**：
1. **承前启后**：U-A 是 N-2 的直接兑付，承接 M4.4 日程骨架最自然；U-B 打通 N-1 让记忆首次带"时间维度"，给 M4.6+ 的长期 arc 打基础
2. **演示价值**：REST 下发临时目标能给 stakeholder 展示一个**交互式** NPC 行为，而非只是"自动生活"
3. **边际收益高**：U-N（soft window）是 0.5 天的小补丁，把 M4.4 N-4 一并清掉不划算也不浪费
4. **风险可控**：三个子项都是"增量 + 回退通道可设"，符合 M4.3 / M4.4 一脉相承的设计风格

**批次划分**：
- **M4.5.0**（U-B + U-N，0.8~1 天）：先做相对轻量的记忆时间感 + soft window，铺垫主动性
- **M4.5.1.a**（U-A 数据层 + REST，0.8 天）：`npc_goal` 建表 + seed + CRUD 接口
- **M4.5.1.b**（U-A plan 节点三路分支 + 前端气泡/列表，1~1.2 天）：`event > goal > schedule` 优先级；前端新增 `SandboxGoals` 小组件（可选）或列入 `Sandbox.vue` 内联展示

## 5. U-A 数据模型草图（Q3=a 情景）

```sql
CREATE TABLE npc_goal (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  npc_id      BIGINT NOT NULL,
  title       VARCHAR(128) NOT NULL COMMENT '目标短文本，直接给 LLM',
  kind        ENUM('scene','player','npc','self') NOT NULL DEFAULT 'player'
              COMMENT '来源：场景事件触发 / 玩家下发 / 其他 NPC 驱动 / 自主产生',
  priority    TINYINT NOT NULL DEFAULT 8 COMMENT '1..10；≥ schedule.priority 即覆盖',
  status      ENUM('active','paused','done','dropped') NOT NULL DEFAULT 'active',
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  expires_at  DATETIME DEFAULT NULL COMMENT '过期自动切 done；NULL 表示手动关闭',
  payload     JSON DEFAULT NULL COMMENT '附加上下文：target_npc_id / target_location / 进度信息',
  INDEX idx_npc_goal_npc_status (npc_id, status),
  INDEX idx_npc_goal_expires (expires_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**Plan 分支伪代码**（Q4=a）：

```ts
// scheduler.tick() NPC loop
const activeGoal = await fetchActiveGoalForNpc(npc.id);  // ORDER BY priority DESC LIMIT 1
const hasEvents = (eventItems?.length ?? 0) > 0;

let planPath: 'event' | 'goal' | 'schedule' | 'idle';
if (hasEvents) {
  planPath = 'event';  // M4.3 行为，不注入 goal/schedule
} else if (activeGoal && activeGoal.priority >= (scheduledActivity?.priority ?? 0)) {
  planPath = 'goal';   // 注入 goal.title 到 system prompt
} else if (scheduledActivity) {
  planPath = 'schedule';  // M4.4 行为
} else {
  planPath = 'idle';
}
nextMeta.plan_path = planPath;  // 前端可视化用
```

## 6. REST 接口草图（Q2=a 情景）

```
POST   /api/engine/goals              # 下发目标
       body: { npc_id, title, kind?, priority?, expires_in_seconds?, payload? }
       resp: { id, ...goal }

PATCH  /api/engine/goals/:id          # 更新状态/优先级
       body: { status? | priority? | expires_at? }
       resp: { id, ...goal }

GET    /api/engine/goals?npc_id=X&status=active
       resp: { items: [...], total }

DELETE /api/engine/goals/:id          # 硬删（status=dropped 的变体）
```

**下发示例**：
```bash
curl -X POST http://localhost:3000/api/engine/goals \
  -H 'Content-Type: application/json' \
  -d '{"npc_id":1,"title":"去图书馆找小美聊一聊","priority":9,"expires_in_seconds":300}'
# → 下一 tick 小明的 plan prompt 将带「【当前目标】去图书馆找小美聊一聊（优先级 9）」
```

## 7. 回退通道（与 M4.3/M4.4 一致）

| 开关 | 默认 | 效果 |
|---|---|---|
| `GOAL_ENABLED` | true | false 时 scheduler 不查 `npc_goal`，plan 回 M4.4 行为 |
| `MEMORY_SLOT_HOUR_ENABLED` | true | false 时 reflect 不注入 slot，`npc_memory.slot_hour` 仍写但不用 |
| `SCHEDULE_SOFT_WINDOW_MIN` | 15 | 设为 0 关闭 soft window，整点硬切 |
