## 0. TL;DR

- **前因**：M4.4 交付日程骨架后，NPC 有 24 小时日程也能接住对话链，但存在 3 类延后（见 `m4.4-summary.md §7` 的 N-1~N-6）：
  1. **缺长期 / 动态驱动力**（N-2）：日程是"硬编码 24 小时"，无**临时目标** / **剧情插入** / **玩家干预**覆盖机制
  2. **记忆无时间感**（N-1）：reflect / summary 写的全是"发生了什么"，没有"我在什么时段做的"
  3. **整点切换突变**（N-4）：`npc_schedule.priority` 存在但 plan 节点从 09:59 → 10:00 会直接硬切，LLM 口吻会突变
- **交付**（roadmap 方案甲 Q1-Q9 全 a 定稿）：
  - **M4.5.0 · U-B + U-N**：reflect 注入【当前时段】+ `npc_memory.slot_hour` 冗余列 + `schedule/softWindow.ts` 15 分钟前瞻（`SIM_CLOCK_MINUTE` 可演示跨边界）
  - **M4.5.1.a · U-A 数据层 + REST**：新表 `npc_goal`；`engine/goal/{config,fetch,crud}.ts` 三件套；`POST / PATCH / GET / DELETE /api/engine/goals` 四个 REST；list/get 入口懒切换 `expires_at` 过期 → `status='done'`
  - **M4.5.1.b · U-A plan 三路 + 前端气泡**：新纯函数 `engine/graph/planPath.ts` 实现 `event > goal > schedule > idle` 四路（goal 仅当 `priority >= scheduledActivity.priority` 才压过）；plan prompt 互斥注入【当前目标】并**关闭**【当前时段计划】；`SimulationMetaV1` 扩 `plan_path` + `active_goal`；前端气泡升四级回退 say > action > **goal** > schedule
- **支线**：M4.5.1.b 前夕用户要求"前台允许 IP 访问"，一并落地 `vite.config.ts` `server.host: true` + `/api` `/uploads` `/ws` 同源代理，前端三条链路改相对路径即可局域网跨设备联调（commit `18b6bf2`）
- **规模**：
  - 后端单测 205 → **251**（+46，超 Q8=a 基线 +31）
  - 前端单测 18 → **23**（+5，超 Q8=a 基线 +3）
  - 新表 **+1**（`npc_goal`）/ 扩列 **+1**（`npc_memory.slot_hour`）
  - 新 env **+5**（`MEMORY_SLOT_HOUR_ENABLED` / `SCHEDULE_SOFT_WINDOW_MIN` / `SIM_CLOCK_MINUTE` / `GOAL_ENABLED` / `GOAL_DEFAULT_TTL_SEC`）
  - 新 REST **+4**（全部落 `/api/engine/goals*`）
  - 新 WS 字段 **+2**（`plan_path` / `active_goal`，沿用 `tick.npc.updated` 不开新通道）
  - 新前端组件 **0**（持续 Q7=a 最小改动）
- **端到端真机冒烟**：脚本按 id DESC 直查 `npc_tick_log` 绕开 tick 历史叠加；goal 下发下一 tick 立即切 `plan_path='goal'` + `active_goal.id/title/priority` 三字段对齐；PATCH done 后下一 tick 回落 `schedule` 且 `active_goal=null`；**11 个断言全 PASS**

## 2. 数据层改动（M4.5 合计）

### 2.1 新表（+1）

```sql
CREATE TABLE npc_goal (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  npc_id      BIGINT NOT NULL COMMENT 'NPC 主键（不加外键，便于冷启动）',
  title       VARCHAR(128) NOT NULL COMMENT '目标短文本，直接注入 plan prompt',
  kind        ENUM('scene','player','npc','self') NOT NULL DEFAULT 'player'
              COMMENT '来源：场景事件 / 玩家下发 / 其他 NPC 驱动 / 自主产生',
  priority    TINYINT NOT NULL DEFAULT 8 COMMENT '1..10；与 schedule.priority 同轴比较',
  status      ENUM('active','paused','done','dropped') NOT NULL DEFAULT 'active',
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  expires_at  DATETIME DEFAULT NULL COMMENT '到期自动切 done；NULL 表示手动关闭',
  payload     JSON DEFAULT NULL COMMENT '附加上下文：target_npc_id / target_location / 进度等',
  INDEX idx_npc_goal_npc_status (npc_id, status),
  INDEX idx_npc_goal_expires (expires_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.2 扩列（+1）

| 表 | 列 | 类型 | 说明 |
|---|---|---|---|
| npc_memory | `slot_hour` | TINYINT NULL | 记忆写入时的时段（0..23），供反思 / RAG 按时段过滤 |

### 2.3 新索引（+3）

- `npc_memory.idx_npc_memory_npc_slot (npc_id, slot_hour)`（复合）
- `npc_goal.idx_npc_goal_npc_status (npc_id, status)`
- `npc_goal.idx_npc_goal_expires (expires_at)`

### 2.4 迁移脚本（+1）

- `backend/src/db/migrate-m45.ts` 一个文件分 2 段幂等执行：
  - 段 1（M4.5.0）：`npc_memory.slot_hour` 列 + 复合索引（列存在跳过 / 索引存在跳过）
  - 段 2（M4.5.1.a）：`npc_goal` 建表 + 两索引（表存在跳过 / 索引存在跳过）

## 3. 环境变量（+5）

```env
# M4.5.0 记忆时间感 + 日程 soft window
MEMORY_SLOT_HOUR_ENABLED=true           # false 时 slot_hour 写 NULL + reflect 不注入【当前时段】
SCHEDULE_SOFT_WINDOW_MIN=15             # 0 关闭；[0,30] 外降级为 0
# SIM_CLOCK_MINUTE=<0..59>              # 可选：演示/单测时注入仿真分钟

# M4.5.1 动态目标系统
GOAL_ENABLED=true                       # false 时 scheduler 不查 npc_goal；REST 写操作 503 GOAL_DISABLED
GOAL_DEFAULT_TTL_SEC=1800               # 默认 expires_in_seconds（未显式传时回落）
```

**回退通道**：

| 开关 | 回退行为 |
|---|---|
| `MEMORY_SLOT_HOUR_ENABLED=false` | slot_hour 列写 NULL、reflect 不注入【当前时段】→ 回 M4.4 行为 |
| `SCHEDULE_SOFT_WINDOW_MIN=0` | 整点硬切，等价 M4.4.1.b 行为 |
| `GOAL_ENABLED=false` | scheduler 不查 `npc_goal`；plan 回 M4.5.0 三路（event/schedule/idle）；REST 写操作 503（GET 仍可读历史） |

## 4. 观测与运维

### 4.1 新增 REST（+4）

```
POST   /api/engine/goals              下发目标
       body: { npc_id, title, kind?, priority?, expires_in_seconds?, payload? }
       resp: { id, ...goal }

PATCH  /api/engine/goals/:id          更新状态/优先级/expires
       body: { status? | priority? | title? | expires_at? | payload? }
       resp: { id, ...goal }

GET    /api/engine/goals?npc_id=X&status=active&page=1&page_size=50
       resp: { items: [...], total, page, page_size }
GET    /api/engine/goals/:id          单条详情

DELETE /api/engine/goals/:id          硬删
```

- 读操作（GET）**不受 `GOAL_ENABLED` 约束**（运维可查历史）
- 写操作（POST/PATCH/DELETE）`GOAL_ENABLED=false` 时返 503 `GOAL_DISABLED`
- list / get 入口触发 `expireLapsedGoals`，把 `active && expires_at<=NOW()` 的行 UPDATE `status='done'`

### 4.2 WS 事件（扩 2 字段）

沿用 M4.3 / M4.4 的 `tick.npc.updated` 通道，不开新事件类型：

```json
{
  "type": "tick.npc.updated",
  "scene_id": 1,
  "tick": 42,
  "npc_id": 1,
  "meta": {
    "plan_path": "goal",
    "active_goal": { "id": 7, "title": "去图书馆找小美", "priority": 9 },
    "scheduled_activity": { "activity": "工作", "location": "书房", "priority": 5 },
    "...": "..."
  }
}
```

### 4.3 结构化日志

- `[goal.fetch] fetchActiveGoalForNpc 失败降级` — DB 异常或表缺字段时降级 null，tick 主链路不阻
- `[goal.crud] expireLapsedGoals 更新 N 行` — 懒切换命中数
- `[schedule.fetch] 查询失败，降级为空日程` — 继承 M4.4

## 7. 已知限制 & 延后项

| # | 描述 | 级别 | 落地节点 |
|---|---|---|---|
| N-1 | scheduler 不主动裁决 goal 完成（`status='done'` 仅靠 REST / `expires_at` 懒切换）；NPC 自身无法"说出完成" | 低 | M4.6：可接 LLM self-report 让 plan / say 输出含 "goal_done" 信号 |
| N-2 | goal 的 `payload` 字段仅用于 REST 查询侧，plan prompt 目前只消费 `title`；`target_npc_id` / `target_location` 不会自动进 prompt | 中 | M4.6 或 M5.x：按 `payload.target_location` 引导 NPC 在 Phaser 里自动移动 |
| N-3 | scene_event → goal 自动派生未实现（Q2=a 计划的"双通道"只兑付 REST 主通道） | 低 | M4.6 或 M5.x |
| N-4 | `plan_path` 经 WS 广播到前端，但前端目前仅把 `active_goal.title` 渲染到气泡；时间线徽章 / 调试面板未呈现 `plan_path` 四色标签 | 低 | M4.6 `Sandbox.vue` 拆分时统一做 |
| N-5 | `npc_goal.priority=0` 的边界情况未显式规避（`computePlanPath` 逻辑仍支持，但 DB 约束是 TINYINT NOT NULL DEFAULT 8，用户可写 0） | 低 | M4.6：可在 controller 层再加 range 校验（1..10）|
| N-6 | live 模式（非 dry_run）的 goal 链路只在 dry_run 冒烟中验证过；真实 LLM 生成样本未跑（避免 token 成本） | 低 | 联调时由实际用户 smoke |

**明确延后**：Sandbox.vue 拆分（U-C，已延后 4 节点，M4.6 强制优先）/ 观测 P95 histogram（U-D）/ LangGraph.js 迁移（U-F）/ 对话 session 聚合（U-Q）/ 玩家插话 @mention（U-R）/ 其他 P2 小优化（U-G ~ U-P）—— 全部归入 M4.6 候选池。
