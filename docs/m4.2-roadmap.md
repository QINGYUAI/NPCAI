## 0. 摘要（TL;DR）

| 目标维度 | M4.1 已交付 | M4.2 要补齐 |
|---|---|---|
| **NPC 思考** | plan → speak → memory 线性图（zod） | **记忆向量检索** + **反思抽象**（目标/情绪/关系） |
| **世界感知** | 只感知到同场景 `neighbors` 名字 | **场景事件**（下雨/打斗/节日）可编程注入 |
| **观察成本** | 后端日志 + 3s 轮询 + cost_usd_total=0 占位 | **WebSocket 推送** + **真实 token/cost 统计** + 前端时间线 |
| **工程债** | `X-Meta-Warn` / budget / meta 前端校验未做 | **0.5 天清零**，不留尾巴 |

**总工作量 ≈ 12.5 天**（单人；含联调与最小测试，不含代码评审 buffer）。

**架构原则**：每个子里程碑都是一次**独立可回滚的纵切**（数据层 → API → 引擎 → 前端 → 测试 → 文档），失败可单独撤销。继续不引入 LangGraph.js —— 保留 M4.3 重构窗口。

## 1. 范围

### 1.1 范围内（M4.2）

1. **M4.2.0 尾补丁**：HTTP `X-Meta-Warn` 头、前端 meta 软校验提示、`ai_config.budget_tokens_per_tick` 字段与校验。
2. **M4.2.1 观测性**：`/ws/engine` WebSocket 事件推送；`ai_call_log` 记录真实 tokens/cost；前端 tick 时间线浮窗。
3. **M4.2.2 记忆向量化 M-1→M-2**：`npc_memory` 新表 + embeddings 检索；plan/speak 节点注入 top-K 相关记忆；`memory_summary` 保留向下兼容。
4. **M4.2.3 反思循环**：每 N tick 触发 reflect 节点；`npc_reflection` 表归档高层抽象；`simulation_meta.plan` 提升稳定性。
5. **M4.2.4 事件总线**：`scene_event` 表 + `POST /api/scene/:id/events` 注入 API；tick 头自动拉事件注入 agent 输入；前端沙盒事件面板（下雨/加 NPC/触发剧情）。

### 1.2 非范围（M4.3+ 再考虑）

- LangGraph.js / LangGraph Python 接管推理图（M4.3）
- Human-in-the-loop（玩家插话、打断、角色扮演）（M4.3）
- 跨场景联动 / 多场景同进程并跑（M4.4）
- 空间网格寻路 / NPC 物理移动（M5 沙盒游戏化）
- 语音/TTS / Avatar 形象生成（M5+）

## 3. 数据层变更

### 3.1 新表（M4.2.2）

```sql
-- M4.2.2 NPC 长期记忆（条目化 + 向量化）
CREATE TABLE IF NOT EXISTS npc_memory (
  id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '主键ID',
  npc_id BIGINT NOT NULL COMMENT 'NPC ID',
  scene_id BIGINT DEFAULT NULL COMMENT '来源场景（可为空，表示全局）',
  tick BIGINT DEFAULT NULL COMMENT '来源 tick',
  type VARCHAR(16) NOT NULL COMMENT 'observation/dialogue/reflection/event',
  content TEXT NOT NULL COMMENT '记忆原文（≤ 1000 字）',
  importance TINYINT DEFAULT 5 COMMENT '1~10 重要度（LLM 或规则打分）',
  embedding JSON DEFAULT NULL COMMENT '向量（float32 array，~1536 维）',
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  last_accessed_at DATETIME(3) DEFAULT NULL COMMENT '上次被检索到的时间',
  access_count INT DEFAULT 0 COMMENT '被检索次数',
  INDEX idx_npc_time (npc_id, created_at),
  INDEX idx_npc_importance (npc_id, importance DESC),
  CONSTRAINT fk_mem_npc FOREIGN KEY (npc_id) REFERENCES npc(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='NPC 长期记忆条目';
```

> **向量检索策略**：MySQL 8.0 原生不支持向量近邻；M4.2.2 先用**客户端余弦**（拉最近 K=200 条记忆到内存算相似度），并配合 `importance` 软过滤。待达到 10k+ 条记忆/NPC 再评估 pgvector / Qdrant 切换（M4.4 候选）。

### 3.2 新表（M4.2.3）

```sql
-- M4.2.3 NPC 反思归档（高层抽象，区别于原始观察）
CREATE TABLE IF NOT EXISTS npc_reflection (
  id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '主键ID',
  npc_id BIGINT NOT NULL COMMENT 'NPC ID',
  scene_id BIGINT NOT NULL COMMENT '场景ID',
  tick BIGINT NOT NULL COMMENT '触发反思的 tick',
  theme VARCHAR(64) DEFAULT NULL COMMENT '反思主题（目标/情绪/关系）',
  content TEXT NOT NULL COMMENT '反思产物',
  source_memory_ids JSON DEFAULT NULL COMMENT '参考的记忆 id 数组',
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  INDEX idx_npc_tick (npc_id, tick DESC),
  CONSTRAINT fk_ref_npc FOREIGN KEY (npc_id) REFERENCES npc(id) ON DELETE CASCADE,
  CONSTRAINT fk_ref_scene FOREIGN KEY (scene_id) REFERENCES scene(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='NPC 反思归档';
```

### 3.3 新表（M4.2.4）

```sql
-- M4.2.4 场景事件（外部注入 / NPC 产生）
CREATE TABLE IF NOT EXISTS scene_event (
  id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '主键ID',
  scene_id BIGINT NOT NULL COMMENT '场景ID',
  type VARCHAR(32) NOT NULL COMMENT '事件类型：weather/dialogue/system/plot',
  actor VARCHAR(64) DEFAULT NULL COMMENT '事件发起者（NPC 名 / system / user）',
  content TEXT NOT NULL COMMENT '事件描述（给 LLM 看的自然语言）',
  payload JSON DEFAULT NULL COMMENT '结构化附加信息（可选）',
  visible_npcs JSON DEFAULT NULL COMMENT 'NULL=全场景可见，数组=指定 NPC 可见',
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  consumed_tick BIGINT DEFAULT NULL COMMENT '首次被 tick 消费时间（供去重）',
  INDEX idx_scene_time (scene_id, created_at DESC),
  CONSTRAINT fk_evt_scene FOREIGN KEY (scene_id) REFERENCES scene(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='场景事件流';
```

## 4. REST / WebSocket API 草案

### 4.1 M4.2.1 WebSocket

```
WS  /ws/engine?scene_id=<id>       订阅指定场景事件
事件类型同 bus：tick.start / tick.npc.updated / tick.end / error
```

**消息格式**：
```json
{ "type": "tick.npc.updated", "scene_id": 1, "tick": 5, "npc_id": 2, "meta": { ... } }
```

前端以此替代 3s 轮询；仍保留 `GET /api/engine/status` 作为 HTTP 降级通道。

### 4.2 M4.2.2 记忆

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/npc/:id/memories?limit=50&type=&since=` | 分页查 NPC 记忆 |
| POST | `/api/npc/:id/memories` | 人工追加记忆（body: `{content, type?, importance?}`） |
| DELETE | `/api/npc/:id/memories/:mid` | 删除单条 |

### 4.3 M4.2.3 反思

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/npc/:id/reflections?limit=20` | 查反思列表 |
| POST | `/api/engine/reflect` `{scene_id, npc_id?}` | 手动触发一次反思（不等 N tick） |

### 4.4 M4.2.4 事件

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/scene/:id/events?limit=50` | 查场景事件流 |
| POST | `/api/scene/:id/events` `{type, content, actor?, visible_npcs?, payload?}` | 注入事件 |
| DELETE | `/api/scene/:id/events/:eid` | 删除事件（调试用） |

## 5. 引擎内部变更

### 5.1 推理图扩展（graph/build.ts）

现有：`plan → speak → memory`  
M4.2 后：`[event-intake] → [memory-retrieve] → plan → speak → memory-store → [reflect?]`

- `event-intake`（M4.2.4）：`SELECT FROM scene_event WHERE scene_id=? AND created_at > NOW()-60s AND consumed_tick IS NULL LIMIT 10` → 注入到 plan 的 user 提示。
- `memory-retrieve`（M4.2.2）：基于 NPC 当前 `plan` 文本做 embedding → 在 `npc_memory` 里 top-K 检索 → 合并 `recent_memories` 字段送入 plan/speak。
- `memory-store`（M4.2.2）：每 tick 把 `latest_say + latest_action` 拼成观察条目 INSERT 到 `npc_memory`（带 embedding）。
- `reflect?`（M4.2.3）：`if tick % REFLECT_EVERY_N_TICK === 0`，启动 reflect 节点拉近 20 条记忆，LLM 生成 3 条 `{theme, content}` 写 `npc_reflection` + 更新 `simulation_meta.plan`。

**每节点仍 zod 严格校验**；重试降级策略不变。

### 5.2 调度器（scheduler.ts）

- M4.2.1：在 `bus.emitEvent` 之后对接 WebSocket 发送器
- M4.2.1：`persistNpcTick` 时从 `ai_call_log` 汇总本 tick 的 `cost_usd_total`（按 npc 累加）
- M4.2.0：`metaStr.byteLength > META_SOFT_BYTES` 时不仅 `console.warn`，还把 warn 添加到 scheduler.status() 里，并在 API 响应加 `X-Meta-Warn: 1` 头
- M4.2.0：tick 前检查 `ai_config.budget_tokens_per_tick`，若上一 tick 超支则本 tick `skipped` 并记事件

### 5.3 llmClient.ts（M4.2.1）

- 调用 LLM 时本地用 `tiktoken` 预估 prompt tokens；拿到响应后读 `json.usage.{prompt_tokens,completion_tokens}`（OpenAI 兼容字段），存进 `ai_call_log`
- 根据 `ai_config.provider + model` 查内建单价表，计算 `cost_usd = prompt_tokens * in + completion_tokens * out`
- 向量化新增封装 `memoryEmbed(npcConfig, text)` 带 30 天磁盘缓存（避免重复 embed 相同文本）

## 9. 风险与降级

| 风险 | 影响 | 缓解 |
|---|---|---|
| embedding API 费用膨胀 | 每 tick 多 N 次 embed 调用 | `memoryEmbed` 加磁盘缓存；`importance < 3` 的记忆不 embed |
| MySQL 内存向量检索在 10k+ 条时变慢 | p99 tick 延迟 | 客户端预取 K=200 热条目；超过阈值报警，文档提示切 pgvector |
| 反思节点输出偏离 JSON schema | tick 标错误 | 复用 M4.1.b 的 zod + retry1 降级 |
| 事件被 LLM 忽略 | 世界感知形同虚设 | prompts 里事件段单独列出并明确"请在 plan 中回应本次事件"；观察小样本后调整 |
| WebSocket 连接风暴 | 后端压力 | 限每场景 8 个订阅；超出 HTTP 降级 |
| 预算限流误伤 | NPC 被静默 | `budget_tokens_per_tick` 默认 2000（远高于当前 ~800 实测）；触发时打 warn 而非直接 skip 首次 |
