## 0. 摘要（TL;DR）

M4.3 完成后，NPC **能互相搭话 + 有 trace 贯穿**，但还缺 3 类能力：

1. **对话链可持续** —— `EVENT_LOOKBACK_SECONDS=60s` + budget skip 会把对话拆断（L-1）
2. **NPC 主动性** —— 当前 NPC 是"响应式"的：只靠当前事件 + 记忆做 plan，**没有日程、没有目标、没有长期动机**
3. **工程可维护性** —— `Sandbox.vue` ≈1800 行（M4.2 X-3 已标延后）；无 P95 observability（X-1）

本节点目标：**从 3 类能力里挑 2~3 条落地**，保持单节点 2~3 天预算。

## 2. 组合方案（4 套）

### 方案甲 · 平衡型（推荐，3 天）

```
T-A（对话链治理）→ T-B（NPC 日程/长期规划）
```

- **逻辑**：先补齐 M4.3 的唯一中级 L-1，再冲 M4.2/M4.3 一直没做的核心能力（NPC 主动性）
- **收益**：节点结束后，NPC 有自发日程 + 对话链不再断，**演示效果跃迁**
- **风险**：T-B 是新能力项，数据层 + plan 前置节点 + 单测量大，测试预算要留 0.5 天 buffer
- **预估**：0.5 + 2.5 = **3 天**（含 live smoke）

### 方案乙 · 稳扎型（1.5 天）

```
T-A + T-C（Sandbox.vue 拆分）+ T-D（反思差异化）
```

- **逻辑**：纯质量工程 + 小能力，不引入新风险
- **收益**：工程可维护性 + 一个小能力补齐
- **风险**：低；`Sandbox.vue` 拆分靠现有 15 前端 tests 兜底
- **预估**：0.5 + 1 + 0.5 = **2 天**

### 方案丙 · 观测强化（2.5 天）

```
T-A + T-E（P95 histogram）+ T-C（Sandbox 拆分）
```

- **逻辑**：所有"可观测"缺口一次补齐，为后续 scale-out 做准备
- **收益**：有了 P95 面板，线上调参有依据（LLM 延迟阈值 / budget 边界）
- **风险**：T-E 需要设计 bucket 边界 + histogram 聚合逻辑，单测量中等
- **预估**：0.5 + 1 + 1 = **2.5 天**

### 方案丁 · 架构重构（3.5 天）

```
T-M（LangGraph.js 接入）+ T-A
```

- **逻辑**：吃掉保留 2 个节点的 LangGraph 窗口，把 `build.ts` 转为可扩展 DAG
- **收益**：为 M4.5+ 的复杂图（条件边、并行、子图）打基础
- **风险**：最大；新依赖 + 全量回归 + 可能牵动 trace/graph 单测；回归 180 单测可能要改 50+ 个
- **预估**：3 + 0.5 = **3.5 天**

### 方案戊 · 大满贯（5~6 天）

```
甲 + T-C + T-E
```

- **逻辑**：能力 + 工程 + 观测一次拉满
- **风险**：工时超单节点惯例（2~3 天），容易拖节奏
- **预估**：5~6 天

## 3. 关键设计决策（Q1~Q9）✅ 全部 CONFIRMED = a

### Q2 · T-A 窗口改法 ✅ **a**

- **a ✅ CONFIRMED**：**混合窗口**：时间窗 `EVENT_LOOKBACK_SECONDS=120s` **或** 条数窗 `EVENT_LOOKBACK_COUNT=50`，任一满足即放行（解 L-1，最柔性）

- **c**：per-dialogue lookback：对话事件走独立 `DIALOGUE_LOOKBACK_SECONDS=300s`，其他事件仍 60s（语义最精准，成本 +0.3 天）

### Q3 · T-B 日程数据粒度 ✅ **a**

- **a ✅ CONFIRMED**：**小时级**（24 槽 / 天），`npc_schedule` 表存 `(npc_id, hour, activity, location, priority)`
- **b**：半小时级（48 槽），精度翻倍但表大小 ×2
- **c**：NPC 自定义槽（`schedule_slots` JSON），最灵活但查询开销大

### Q4 · T-B 日程驱动的 plan 集成点 ✅ **a**

- **a ✅ CONFIRMED**：**plan 节点前置增强**：有 event 走事件驱动（现状），无 event 走日程驱动（新）；两路径互斥

### 7.2 M4.4.1 · T-B NPC 日程 / 长期规划（2.5 天）

#### 7.2.1 数据层（迁移 `migrate-m44.ts` 步骤 2）

```sql
CREATE TABLE IF NOT EXISTS npc_schedule (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  npc_id      INT NOT NULL,
  hour        TINYINT NOT NULL,            -- 0..23
  activity    VARCHAR(128) NOT NULL,
  location    VARCHAR(64) DEFAULT NULL,
  priority    TINYINT DEFAULT 5,           -- 1..10
  updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_npc_hour (npc_id, hour),
  INDEX idx_npc_schedule_npc (npc_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**Seed 脚本**（迁移末尾）：为当前 DB 中所有 NPC 各生成 24 行默认模板（6~8 点早餐、8~12 工作、12 午餐、13~17 工作、18 晚餐、19~22 休闲、23~5 睡眠）。幂等：`INSERT ... ON DUPLICATE KEY UPDATE activity=VALUES(activity)`。
