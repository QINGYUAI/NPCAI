## 0. 摘要（TL;DR）

M4.3 完成后，NPC **能互相搭话 + 有 trace 贯穿**，但还缺 3 类能力：

1. **对话链可持续** —— `EVENT_LOOKBACK_SECONDS=60s` + budget skip 会把对话拆断（L-1）
2. **NPC 主动性** —— 当前 NPC 是"响应式"的：只靠当前事件 + 记忆做 plan，**没有日程、没有目标、没有长期动机**
3. **工程可维护性** —— `Sandbox.vue` ≈1800 行（M4.2 X-3 已标延后）；无 P95 observability（X-1）

本节点目标：**从 3 类能力里挑 2~3 条落地**，保持单节点 2~3 天预算。

## 2. 组合方案（4 套）

### 方案甲 · 平衡型（推荐，3 天）

```
T-A（对话链治理）→ T-B（NPC 日程/长期规划）
```

- **逻辑**：先补齐 M4.3 的唯一中级 L-1，再冲 M4.2/M4.3 一直没做的核心能力（NPC 主动性）
- **收益**：节点结束后，NPC 有自发日程 + 对话链不再断，**演示效果跃迁**
- **风险**：T-B 是新能力项，数据层 + plan 前置节点 + 单测量大，测试预算要留 0.5 天 buffer
- **预估**：0.5 + 2.5 = **3 天**（含 live smoke）

### 方案乙 · 稳扎型（1.5 天）

```
T-A + T-C（Sandbox.vue 拆分）+ T-D（反思差异化）
```

- **逻辑**：纯质量工程 + 小能力，不引入新风险
- **收益**：工程可维护性 + 一个小能力补齐
- **风险**：低；`Sandbox.vue` 拆分靠现有 15 前端 tests 兜底
- **预估**：0.5 + 1 + 0.5 = **2 天**

### 方案丙 · 观测强化（2.5 天）

```
T-A + T-E（P95 histogram）+ T-C（Sandbox 拆分）
```

- **逻辑**：所有"可观测"缺口一次补齐，为后续 scale-out 做准备
- **收益**：有了 P95 面板，线上调参有依据（LLM 延迟阈值 / budget 边界）
- **风险**：T-E 需要设计 bucket 边界 + histogram 聚合逻辑，单测量中等
- **预估**：0.5 + 1 + 1 = **2.5 天**

### 方案丁 · 架构重构（3.5 天）

```
T-M（LangGraph.js 接入）+ T-A
```

- **逻辑**：吃掉保留 2 个节点的 LangGraph 窗口，把 `build.ts` 转为可扩展 DAG
- **收益**：为 M4.5+ 的复杂图（条件边、并行、子图）打基础
- **风险**：最大；新依赖 + 全量回归 + 可能牵动 trace/graph 单测；回归 180 单测可能要改 50+ 个
- **预估**：3 + 0.5 = **3.5 天**

### 方案戊 · 大满贯（5~6 天）

```
甲 + T-C + T-E
```

- **逻辑**：能力 + 工程 + 观测一次拉满
- **风险**：工时超单节点惯例（2~3 天），容易拖节奏
- **预估**：5~6 天

## 3. 关键设计决策（Q1~Q9）

> **每问给出 a/b/c 三个方向，a 一般为 roadmap 默认倾向；评审阶段不动代码，全部确认后再进实施。**

### Q1 · 范围（最关键）

- **a**：**方案甲**（T-A + T-B，3 天，能力跃迁）
- **b**：方案乙（T-A + T-C + T-D，2 天，稳扎）
- **c**：方案丙（T-A + T-E + T-C，2.5 天，观测强化）
- **d**：方案丁（T-M + T-A，3.5 天，架构重构）
- **e**：方案戊（大满贯，5~6 天）
- **f**：自定义组合（你自己挑 ID，如"T-A + T-E + T-L"）

### Q2 · T-A 窗口改法

- **a**：**混合窗口**：时间窗 `EVENT_LOOKBACK_SECONDS=120s` **或** 条数窗 `EVENT_LOOKBACK_COUNT=50`，任一满足即放行（解 L-1，最柔性）
- **b**：只调大默认值到 180s，不引入条数窗（最小改动）
- **c**：per-dialogue lookback：对话事件走独立 `DIALOGUE_LOOKBACK_SECONDS=300s`，其他事件仍 60s（语义最精准，成本 +0.3 天）

### Q3 · T-B 日程数据粒度（若 Q1 选 a / e）

- **a**：**小时级**（24 槽 / 天），`npc_schedule` 表存 `(npc_id, hour, activity, location, priority)`
- **b**：半小时级（48 槽），精度翻倍但表大小 ×2
- **c**：NPC 自定义槽（`schedule_slots` JSON），最灵活但查询开销大

### Q4 · T-B 日程驱动的 plan 集成点（若 Q1 选 a / e）

- **a**：**plan 节点前置增强**：有 event 走事件驱动（现状），无 event 走日程驱动（新）；两路径互斥
- **b**：plan 里并行注入"当前日程 hint"，让 LLM 自己权衡（可能与事件冲突）
- **c**：新 `schedule` 节点独立，在 plan 前判定，产出 `scheduled_activity` 写入 `EventBlockItem`（统一数据流）

### Q5 · LangGraph.js 是否本节点进（独立于 Q1）

- **a**：**本节点不进**，继续保留到 M4.5+（避免和 T-B 新能力撞车）
- **b**：本节点进，但仅替换 `build.ts` 外壳，不扩展子图（最小风险路径）
- **c**：本节点进且配合 T-B 一起做（T-B 的新节点直接用 LangGraph 定义）

### Q6 · 迁移策略

- **a**：**单文件 `migrate-m44.ts`**，继续幂等；本节点若有新表（`npc_schedule`）也塞里面
- **b**：T-B 新表独立一个迁移文件（清晰度优先）
- **c**：不新增迁移（若本节点仅选方案乙/丙）

### Q7 · 前端策略

- **a**：**最小改动**：若选 T-B，仅在状态气泡多一行 `📅 当前日程: <activity>`，不新增组件
- **b**：新增日程可视化抽屉（新组件，复用 `SandboxEvents.vue` 样式）
- **c**：若选 T-C，优先完成拆分后再考虑日程 UI

### Q8 · 回归单测门槛

- **a**：**后端 ≥ 200（+20）**，前端 ≥ 18（+3）；方案甲为准
- **b**：后端 ≥ 210（+30），更严
- **c**：仅要求全绿，不设数量门槛

### Q9 · 子节点拆分粒度（若选方案甲）

- **a**：**2 子节点**：M4.4.0（T-A）+ M4.4.1（T-B 含数据层 + plan 集成 + UI）
- **b**：3 子节点：M4.4.0（T-A）+ M4.4.1.a（T-B 数据层 + schedule 节点）+ M4.4.1.b（T-B plan 集成 + UI）
- **c**：4 子节点（最细）：M4.4.0 / M4.4.1.a 数据层 / M4.4.1.b 节点集成 / M4.4.1.c UI

## 5. 风险与降级

| 风险 | 概率 | 降级 |
|---|---|---|
| T-B 日程节点引入新 LLM 调用，预算超支 | 中 | 日程走模板而非 LLM（简单匹配"当前 hour → activity"） |
| T-A 混合窗口引入后老事件涌入 intake prompt 膨胀 | 低 | 新增 `EVENT_MAX_BLOCK_ITEMS=30` 上限兜底 |
| T-B 新表 `npc_schedule` 初始数据缺 | 高 | seed 脚本同迁移一起跑，默认给 4 NPC 各 24h 模板 |
| Sandbox 拆分（方案乙/丙）回归 | 中 | 拆分前先跑全 frontend test 基线，拆分后逐组件 diff |
