# M4.5 交付展示：让 NPC 有目标，有时间感，也能平滑过渡

> 版本 1.0 · 2026-04-24 · 关盘 tag `v0.4.5-closed`
> 详尽工程记录见 [`m4.5-summary.md`](./m4.5-summary.md) · 上一节点 [`m4.4-showcase.md`](./m4.4-showcase.md)

---

## 一句话

**以前**：NPC 严格按 24 小时日程表生活；十点一到就机械切换"工作 @ 书房"，玩家/剧情想临时打断它、让它去干点别的，没有办法。
**现在**：**玩家 / 剧情可以随时给 NPC 下一个"临时目标"**，NPC 下一 tick 立即放下日程去推进目标；目标完成 / 过期后，又自然回到日程；而且所有过渡都是"提前 15 分钟软切换"，不再有整点突变。

---

## 看得见的变化

### 1. 随时给 NPC 下一个临时目标

以前：NPC 10 点固定"工作@书房"，不能打断。
现在：一条 `curl` 就能插入目标，下一 tick NPC 立即响应。

```bash
curl -X POST http://localhost:3000/api/engine/goals \
  -H 'Content-Type: application/json' \
  -d '{
    "npc_id": 1,
    "title": "去图书馆找小美聊一聊",
    "priority": 9,
    "expires_in_seconds": 300
  }'
```

下一 tick 的 NPC 气泡：

```
🎯 目标: 去图书馆找小美聊一聊
```

plan prompt 里同时发生了：
- 原本的「【当前时段计划】工作 at 书房」**被自动关闭**
- 替换为「【当前目标】去图书馆找小美聊一聊（请优先推进该目标；结合角色性格决定具体做法）」

这让 LLM 能专注推进目标，而不是同时在"工作"和"去找小美"之间纠结。

### 2. 目标完成后自然回到日程

目标达成或过期后，状态机如下：

```
tick  N   |  💬 说："那我先过去图书馆"    ← goal 活跃，plan_path='goal'
tick N+1  |  🎯 目标: 去图书馆找小美聊一聊
         → 玩家 PATCH status=done
tick N+2  |  📅 当前日程: 工作 @ 书房    ← plan_path='schedule'，active_goal=null
```

- `PATCH /api/engine/goals/:id { "status": "done" }` 立即关闭
- 或者 `expires_at` 自动过期（下次查询时懒切换）

### 3. 优先级同轴比较

如果当前日程 `priority=8`（比如"重要会议"），而玩家下发 `priority=5` 的目标（比如"顺便泡个咖啡"），**日程胜出**，NPC 不会被轻易打断。

```
goal.priority=5  vs  schedule.priority=8   → plan_path='schedule'（日程赢）
goal.priority=9  vs  schedule.priority=5   → plan_path='goal'（目标赢）
goal.priority=5  vs  schedule.priority=5   → plan_path='goal'（同值时目标赢，表达"主动性"）
```

### 4. 整点切换不再突变（soft window）

M4.4 里 09:59 → 10:00 是"硬切"：一秒钟前还在"早餐@厨房"，一秒钟后突然变"工作@书房"。
M4.5 引入 15 分钟前瞻：

```
09:44  → 仍按 09:00 时段（早餐@厨房）
09:45  → 已按 10:00 时段（工作@书房）    ← 提前 15 分钟进入下一时段
23:50  → 00:00 时段（跨日绕环正常）
```

环境变量 `SCHEDULE_SOFT_WINDOW_MIN` 可调，默认 15，设 0 回 M4.4 硬切行为。

### 5. 反思带"时间感"了

M4.4 里反思只关心"发生了什么"，读起来像流水账。
M4.5 让反思 prompt 带上【当前时段】，memory 表也增加了 `slot_hour` 列：

```
【当前时段】15:00-16:00 吃下午茶 @ 阳台

请基于本 tick 的行为与对话，生成一条简短反思……
```

这让长期记忆第一次有了"我在下午茶时间聊过这件事"这种时间维度，为 M4.6+ 的长期 arc 打基础。

### 6. 前端沙盒气泡四级回退

气泡优先级（从高到低）：

| 优先级 | 条件 | 内容 |
|---|---|---|
| 1 | NPC 刚说话 | `💬 说："……"` |
| 2 | NPC 在做动作 | `🚶 动作: ……` |
| **3（新）** | **有活跃目标** | **`🎯 目标: <title>`** |
| 4 | 闲着 + 有日程 | `📅 当前日程: <activity> @ <location>` |

## 后端做了什么

### 新表：`npc_goal`

```sql
CREATE TABLE npc_goal (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  npc_id      BIGINT NOT NULL,
  title       VARCHAR(128) NOT NULL,
  kind        ENUM('scene','player','npc','self') DEFAULT 'player',
  priority    TINYINT DEFAULT 8,
  status      ENUM('active','paused','done','dropped') DEFAULT 'active',
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  expires_at  DATETIME DEFAULT NULL,
  payload     JSON DEFAULT NULL,
  INDEX (npc_id, status),
  INDEX (expires_at)
);
```

### 新 REST

```
POST   /api/engine/goals          下发目标
PATCH  /api/engine/goals/:id      改状态/优先级/过期
GET    /api/engine/goals?...      列表（分页 + npc_id / status 过滤）
GET    /api/engine/goals/:id      单条
DELETE /api/engine/goals/:id      硬删
```

### 新纯函数：`computePlanPath`

```ts
// backend/src/engine/graph/planPath.ts
export type PlanPath = 'event' | 'goal' | 'schedule' | 'idle';

export function computePlanPath(input): PlanPath {
  if (input.hasEvents) return 'event';          // 有事件优先响应（M4.3 行为）
  if (input.activeGoal &&
      input.activeGoal.priority >= (input.scheduledActivity?.priority ?? 0)) {
    return 'goal';                               // 目标压过日程
  }
  if (input.scheduledActivity) return 'schedule';
  return 'idle';
}
```

这个决策纯粹、可单测、未来迁 LangGraph.js 时 0 成本移植。

### WS 扩字段

沿用 `tick.npc.updated` 通道，`meta` 里新增两个字段：

```json
{
  "plan_path": "goal",
  "active_goal": { "id": 7, "title": "去图书馆找小美", "priority": 9 }
}
```

前端取 `meta.active_goal.title` 就能渲染气泡。

## 前端做了什么

### 零新组件（持续 Q7=a 最小改动策略）

仅改 2 处：

1. `frontend/src/utils/sandbox.ts::extractBubbleText` 签名扩 `activeGoal?` 参数，插入第三级回退
2. `frontend/src/components/Sandbox.vue::refreshBubbles` 从 `n.simulation_meta.active_goal` 取值

### 支线：允许 IP 访问

`frontend/vite.config.ts` 一次改动，手机 / iPad 通过局域网 IP 就能全链路使用：

```ts
export default defineConfig({
  server: {
    host: true,              // 监听 0.0.0.0，局域网可访问
    allowedHosts: true,      // 不校验 Host 头，LAN IP 直通
    proxy: {
      '/api':     { target: BACKEND_ORIGIN, changeOrigin: true },
      '/uploads': { target: BACKEND_ORIGIN, changeOrigin: true },
      '/ws':      { target: BACKEND_WS,     ws: true, changeOrigin: true },
    },
  },
});
```

三条链路（REST API / 静态上传 / WebSocket）统一走 Vite 同源代理，前端代码全改相对路径即可。

## 回退通道（全在环境变量里）

| 开关 | 默认 | 关闭后的效果 |
|---|---|---|
| `GOAL_ENABLED` | true | scheduler 不查 `npc_goal`，plan 回 M4.5.0 三路；REST 写操作 503 |
| `MEMORY_SLOT_HOUR_ENABLED` | true | slot_hour 列写 NULL + reflect 不注入【当前时段】 |
| `SCHEDULE_SOFT_WINDOW_MIN` | 15 | 设 0 回 M4.4 硬切 |

所有开关支持热切换：改 env 后**下一 tick 立即生效**，无需重启服务。

## 典型演示剧本（~3 分钟）

1. 启动引擎（dry_run 模式，避免 LLM 成本）：
   ```bash
   curl -X POST http://localhost:3000/api/engine/start \
     -H 'Content-Type: application/json' \
     -d '{"scene_id":1,"dry_run":true,"interval_ms":2000,"max_ticks":60}'
   ```
2. 观察气泡——NPC 按日程走，闲时显示 `📅 当前日程: …`
3. **给 NPC 下临时目标**：
   ```bash
   curl -X POST http://localhost:3000/api/engine/goals \
     -H 'Content-Type: application/json' \
     -d '{"npc_id":1,"title":"去图书馆找小美聊一聊","priority":9,"expires_in_seconds":300}'
   ```
4. 下一 tick：气泡切 `🎯 目标: 去图书馆找小美聊一聊`；前端 / WS / REST 三处的 `simulation_meta.plan_path='goal'`
5. **完成目标**：
   ```bash
   curl -X PATCH http://localhost:3000/api/engine/goals/<id> \
     -H 'Content-Type: application/json' \
     -d '{"status":"done"}'
   ```
6. 下一 tick：气泡回 `📅 当前日程: …`；`plan_path='schedule'`、`active_goal=null`
