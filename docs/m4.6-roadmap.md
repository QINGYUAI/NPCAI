## 0. 背景与总体约束

### 0.1 M4.5 关盘后的坐标

- **M4.5.0 U-B/U-N** 把"记忆时间感"和"日程软窗口"补齐；**M4.5.1.a/b U-A** 把"动态目标覆盖"从零做到生产可用（REST + plan 三路 + 前端气泡）
- 现场演示链路：玩家通过 REST 下 goal → NPC 下一 tick `plan_path='goal'` + 【当前目标】prompt → goal 完成 / 过期后回落 schedule
- **短板**（M4.5 关盘 summary §7 提出 + 本轮 C 阶段实测揭示）：
  1. **玩家没有对话入口**：现在玩家下 goal 是"写数据库"，不是"说话"；NPC 之间会彼此 @对方的 dialogue event，但玩家说话没法触达特定 NPC（U-R 自 M4.3 延后至今 **3 节点**）
  2. **对话缺会话元数据**：turn-taking 靠 `parent_event_id` 单链 + `conv_turn` 编号，**没有"会话"这个一等概念**，前端无法按"一次聊天"为单位展示 / 统计 / 归档（U-Q 自 M4.3 延后 **3 节点**）
  3. **goal 的"event 抢占"没有运维视角**：本轮 C1 live 实测发现 dialogue event 连续生成时 goal(priority=9) 永远没机会命中；设计正确，但运维侧看不见"现在 event 队列压着什么"（**C-new-1** 本轮新增）
  4. **Sandbox.vue 1800 行单文件** 已延后 **4 节点**，再不拆会成为 M4.6 新功能的硬约束（U-C）

### 0.2 M4.6 的定位

- **主题**：**玩家交互深化** — 玩家第一次能直接"@NPC 说话"并看到"一次完整会话"
- **不做**：LangGraph.js 迁移（U-F 继续延，等 M4.7+）/ 多 agent 并行（远期）/ 玩家多人同屏（远期）
- **选做（看 Q1 定）**：U-C Sandbox 拆分（已延后过多）、U-D P95 histogram 小窗口、N-5 priority range 校验

## 2. 组合方案（5 套）

### 方案甲 · 玩家交互深化（推荐，2.3~2.8 天）

```
U-R（@mention 路由）→ U-Q（conversation 聚合表）→ N-5（priority range 校验，收尾小项）
```

- **逻辑**：一次性把"玩家说话"（对话入口）和"会话"（对话元数据）两个长期欠账补齐；NPC 之间已有的 dialogue event 链天然兼容新表（只加 `conversation_id` 列），零中断
- **收益**：
  - 玩家第一次可以**直接跟 NPC 对话**（不再仅能写 goal）
  - 前端抽屉按"会话"分组展示，UX 对标成熟产品（ChatGPT 会话列表）
  - 为 M5 多人玩家 / UGC 剧情编辑器铺直接路径
- **风险**：
  - U-Q 新表 + 历史 scene_event 回填 `conversation_id`（迁移脚本需写幂等/分批，避免锁表）
  - U-R 改 `plan prompt` 分支多（被 @ / 未被 @ / 多人同聊）
- **预估**：1 + 1.5 + 0.2 = **2.7 天**（含 live smoke，不含 buffer）

### 方案乙 · goal 系统全功能（2.5~3.2 天）

```
N-1（LLM self-report goal_done）→ N-2（goal→Phaser 移动）→ N-3（scene_event→goal 派生）
```

- **逻辑**：把 M4.5 的 goal 从"REST 数据层能用"跃迁到"NPC 能自主推进 + 能驱动画面 + 能被场景自动触发"三件套
- **收益**：演示效果最强，goal 系统从"写 DB 能看 prompt"变成"真实能看见 NPC 移动去完成"
- **风险**：N-2 依赖 Phaser 路径规划（当前仅坐标跳转）；N-1 要让 LLM 输出稳定 `goal_done` 字段（prompt 约束 + 解析容错）
- **预估**：0.8 + 1 + 0.8 + 0.3 buffer = **2.9 天**

### 方案丙 · 工程债务清理（2.3~2.8 天）

```
U-C（Sandbox.vue 拆分）+ U-D（观测 P95）+ N-5（priority 校验） + C-new-1（event 抢占埋点）
```

- **逻辑**：集中清延后 4 节点的 U-C 和延后 3 节点的 U-D；**0 新功能、0 用户可见变化**
- **收益**：为 M4.7/M5 拆分空间 + 观测抓手；U-D 还能回答"本节点交付实际 P95 是多少"
- **风险**：纯工程不出戏；U-C 拆分涉及 1800 行，回归面大
- **预估**：1 + 1 + 0.2 + 0.5 = **2.7 天**

### 方案丁 · 混合（一个强制优先 + 一个演示点 + 一个观察点，2.5~3 天）

```
U-C（Sandbox 拆分）+ N-1（goal 自主裁决）+ U-D（P95 小窗口）
```

- **逻辑**：一次吃掉一个强制优先 + 一个用户可见的演示点 + 一个观测抓手
- **收益**：均衡打法；Sandbox 拆分不再延后，N-1 提升 goal 演示闭环，P95 补齐 M4.2 老账
- **风险**：三件事独立，buffer 吃紧；U-C 拆分 + N-1 prompt 改动并行有小回归风险
- **预估**：1 + 0.8 + 1 + 0.2 buffer = **3.0 天**

### 方案戊 · 架构跃迁（3~3.5 天，高风险）

```
U-F（LangGraph.js 迁移）+ U-L（conv_turn 兜底）+ U-O（LLM hint 通道）
```

- **逻辑**：把 plan/goal/schedule 三路 DAG 化，为 M5 多 agent 并行 / 分支留空间
- **风险**：**无 buffer**；251 + 36 单测全量回归；LangGraph.js 学习曲线
- **预估**：2.5 + 0.3 + 0.3 = **3.1 天**

## 3. 关键设计决策（Q1~Q9）

> 评审阶段请对每条给出 a/b/c 选项；v0.2 定稿后锁定。

### Q1 · 整体范围（主线）

- **a**：方案甲 U-R + U-Q + N-5（玩家交互深化，2.7 天，推荐）
- **b**：方案乙 N-1 + N-2 + N-3（goal 系统全功能，2.9 天，演示效果最强）
- **c**：方案丙 U-C + U-D + N-5 + C-new-1（工程债务清理，2.7 天）
- **d**：方案丁 U-C + N-1 + U-D（混合，3.0 天）
- **e**：方案戊 U-F + U-L + U-O（LangGraph 迁移，3.1 天，高风险）

### Q2 · U-R @mention 路由形态（若 Q1=a/d）

- **a**：**前端显式字段** — 玩家发消息时必须选目标 NPC；`scene_event.payload.target_npc_id=X`；plan prompt 读 target（简单可靠）
- **b**：**LLM 自动识别** — 玩家自由文本里出现 "@NPC_NAME" → 后端正则解析 → target_npc_id（UX 好，但解析歧义）
- **c**：**双通道** — UI 选 + 文本 @ 都支持，UI 选优先

### Q3 · U-Q `conversation` 数据模型（若 Q1=a）

- **a**：**最小**：`(id, scene_id, root_event_id, last_turn_at, status ENUM('active','ended'))`，participants 靠 JOIN event 计算
- **b**：**标准**：在 a 基础上加 `participants JSON`、`turn_count INT`、`last_content VARCHAR(128)`（列表渲染时省 JOIN）
- **c**：**丰富**：在 b 基础上再加 `topic VARCHAR(64) NULL`（LLM 摘要）、`avg_interval_ms INT`（分析用）

### Q4 · U-Q 会话生命周期

- **a**：**自动**：基于 `EVENT_LOOKBACK_SECONDS` 窗口；lookback 内有新 event 则 `status=active`，超时自动 `ended`（简单）
- **b**：**LLM 识别**：prompt 输出 "conversation_done" 信号触发 `ended`
- **c**：**自动 + 玩家手动**：玩家可点"结束会话"强制归档（最接近产品直觉）

### Q5 · U-C Sandbox.vue 拆分（若 Q1 包含 U-C）

- **a**：**推迟** — Q1 非方案丙/丁时，U-C 继续延（第 5 次）
- **b**：**局部**：仅拆 conversation / @mention 相关代码为 `SandboxConversation.vue`，其他 5 组件留 M4.7
- **c**：**完整**：一次拆 6 组件（成本 1 天，风险中等）

### Q6 · 前端改动尺度

- **a**：**最小** — 复用现有组件，仅在输入区加 NPC 下拉 + 会话按会话在抽屉里分组（无新组件）
- **b**：**中等** — 新 `MentionPicker.vue` + `ConversationDrawer.vue` 两个小组件
- **c**：**较大** — 新 MentionPicker / ConversationDrawer / SandboxGoals 三组件（含 N-4 调试抽屉徽章）

### Q7 · 迁移组织

- **a**：单文件 `migrate-m46.ts`（历史惯例，M4.2~M4.5 均如此）
- **b**：分节 `migrate-m46-{0|1}.ts`（若子节点多且改表冲突）

### Q8 · 测试基线

- **a**：后端 ≥266（+15）/ 前端 ≥44（+8）
- **b**：后端 ≥275（+24）/ 前端 ≥50（+14）

### Q9 · 子节点数

- **a**：**2 批次** — M4.6.0 = U-Q（表 + 历史回填 + 前端列表），M4.6.1 = U-R（@mention + plan 分支 + N-5）
- **b**：**3 批次** — M4.6.0 = U-Q 数据层，M4.6.1 = U-Q 前端，M4.6.2 = U-R
- **c**：**1 批次** — 不分，整体推进

## 5. 数据模型草图（方案甲）

```sql
-- [M4.6.0] 对话会话聚合
CREATE TABLE conversation (
  id               BIGINT PRIMARY KEY AUTO_INCREMENT,
  scene_id         BIGINT NOT NULL,
  root_event_id    BIGINT NOT NULL COMMENT '会话第一条 scene_event.id',
  participants     JSON   NOT NULL COMMENT '[npc_id, ...]；玩家按需用 0 或 -1 占位',
  last_turn_at     DATETIME NOT NULL,
  turn_count       INT NOT NULL DEFAULT 1,
  status           ENUM('active','ended') NOT NULL DEFAULT 'active',
  ended_at         DATETIME DEFAULT NULL,
  created_at       DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_conversation_scene_last (scene_id, last_turn_at DESC),
  INDEX idx_conversation_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
  COMMENT='[M4.6.0] 对话会话聚合表（由 scheduler 自动开闭）';

-- [M4.6.0] scene_event 归属会话
ALTER TABLE scene_event
  ADD COLUMN conversation_id BIGINT DEFAULT NULL
    COMMENT '所属会话 id（可 NULL：手插系统事件、历史数据未回填）',
  ADD INDEX idx_scene_event_conversation (conversation_id);

-- 历史回填（分批 1000 行）：按 parent_event_id 追溯根节点 → 同根节点归一个 conversation
-- 无 parent 的事件各自开新会话
```

## 8. 风险 & 缓解

| 风险 | 缓解 |
|---|---|
| 历史 scene_event 回填锁表 | 分批 1000 行 + `SELECT FOR UPDATE SKIP LOCKED`；大批量时可在维护窗口执行 |
| @mention 前端 NPC 多选 UX 复杂 | Q6=a 仅允许单选（与 dialogue.event visible_npcs=1 保持一致）；多人 @ 留到 M4.7 |
| `plan_path='mentioned'` 与 event 冲突 | 顺序固定 `mentioned > event`；被 @的 NPC 即使有其他 event 也优先回应玩家 |
| `conversation_id` 外键未加 → 数据不一致 | 加定期巡检 SQL：`SELECT COUNT(*) FROM scene_event WHERE conversation_id NOT IN (SELECT id FROM conversation)` 应恒为 0 |

### 12.3 观察期验收（人工）

运行 `npm run smoke:live-goal -- --scene=1 --npc=1` 时，期望观察到以下三种结果之一：

| 结果 | 代表性输出 | 对应结论 |
|---|---|---|
| A. PASS | `闭环：tick#X(goal) → PATCH done → tick#Y(schedule/idle)` | goal 路径工程正常，M4.5 稳定 |
| B. 超时 + 事件抢占 hint | `✗ 超时未观察到 plan_path='goal'；最近一条 path=event` | 设计正确（event > goal），但业务侧需要运维观察窗口（对应延后项 C-new-1） |
| C. 超时 + 日程抢占 | `最近一条 path=schedule`（priority 低） | goal.priority=9 低于日程 priority；加大到 10 或检查 schedule 数据 |

观察期不设置硬性时长，由用户在日常使用中累积现场反馈。结束方式：用户显式指示"可以重评 M4.6"。
