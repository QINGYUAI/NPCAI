## 0. 范式对齐（主要目的）

本需求的**主要目的**与下列公开资料所描述的体系一致：**多智能体在虚拟小镇/环境中的可信行为、社会互动与叙事涌现**——其学术源头为斯坦福论文 *Generative Agents: Interactive Simulacra of Human Behavior*（生成式智能体：人类行为的交互式拟像），常被通俗称为「**斯坦福 AI 小镇**」。

- **中文导读（用户指定参考）**：[智能体之斯坦福 AI 小镇（Generative Agents: Interactive Simulacra of Human Behavior）](https://blog.csdn.net/qq_52053775/article/details/139194177)（CSDN）
- **论文核心思想（与产品语义的映射）**：
  - **人物（智能体）**：具有身份、目标与口吻的个体；论文中以「记忆流—反思—计划」驱动行为；在本系统中 **以现有「角色 NPC」为主载体**（人设、系统提示、所用模型等），对应「小镇里的居民」。
  - **场景（环境/情境）**：智能体活动与相遇的**空间或社会情境**（如居所、工作点、公共活动）；不一定是像素画面，而是 **可编排的「何地、何人、何上下文」**，对应论文中的**环境与交互背景**。
  - **关系**：多智能体可在同一场景内被编排，为后续实现「沟通、协作、日程类行为」或对接外部模拟器提供**数据层**。

> **范围说明**：论文中的**完整运行时**（沙盒引擎内自动执行、完整记忆流引擎等）仍可能由外部承担。本文档 **v0.x 聚焦数据与 API**：**人物档案 + 场景 + 多对多关联**；并在 `npc` 上 **预留 JSON 字段** 存放外部仿真回写的记忆/反思摘要（见 §6.3、§10）。**承载上述能力的主引擎（见 §1.4）目前尚未选定**，与 AINPC 本仓库解耦推进。

## 1. 背景与目标

### 1.1 项目现状（AINPC）

- **AI 配置**：多模型连接与参数管理（表 `ai_config`，前端「AI 配置」Tab）。
- **角色 NPC**：名称、人设、系统提示、关联模型、头像等（表 `npc`，前端「角色 NPC」Tab）。
- **能力边界**：当前数据模型以「单表 NPC + 外键到 ai_config」为主，**没有独立的「场景」实体**；NPC 与「小镇中的空间/情境」之间无结构化关联，难以按 **Generative Agents** 思路做「**谁在何地、与谁同场**」的编排。

### 1.2 建设目标

在 **AINPC 本仓库** 内增加 **人物** 与 **场景** 相关能力，使运营或创作者能够：

1. **人物（生成式智能体档案）**：在现有 NPC 基础上，明确其作为 **小镇/仿真中的智能体** 的身份；沿用 `npc` 表承载核心字段，便于与 §0 所述范式及外部工具对齐。
2. **场景（环境/情境）**：引入可管理的 **场景** 实体，表示 **地点、聚会、剧情段或会话上下文**；通过 **场景 ↔ NPC** 多对多关联，表达「**哪些智能体出现在同一场**」，为涌现式互动与后续扩展提供基础数据。

> 说明：本文档中的「场景」是 **叙事/空间/社会情境数据**。**可视化与互动表现**：可参考开源项目 [pixel-agents](https://github.com/pablodelucca/pixel-agents) 中「人物在场景中活动、状态反馈、多代理同屏」等**交互思路**作灵感；**不与该项目做技术联动或集成**，若存在更合适的方案可完全采用其它实现（见 §10、§12）。

### 1.3 成功标准（验收方向）

- 用户可在界面中 **创建、编辑、删除、列表筛选** 场景。
- 用户可将 **一个或多个 NPC（人物）** 关联到某场景；**首版**以场景侧维护关联为主；**NPC 侧展示/跳转所属场景**见 §4.3（后续迭代）。
- 后端提供 **稳定 REST API**，与现有 `/api/npc`、`/api/config` 风格一致（统一 `code`/`message`/`data` 约定）。
- 数据库变更可 **迁移**（提供迁移脚本或增量 SQL），不破坏已有 `npc` / `ai_config` 数据。
- **（范式向）** 数据模型与文案上可自然延伸到「多智能体 + 多场景」编排，不为后续接入记忆/计划模块制造结构性障碍。

### 1.4 主引擎（仿真 / 运行时）— **未选定**

此处 **「主引擎」** 指：驱动「斯坦福 AI 小镇」式体验的 **仿真与运行时**——包括但不限于：多智能体 **时间推进**、**记忆流** 的写入与检索、**反思 / 计划** 循环、与环境或其它智能体的 **交互调度**，以及可选的 **可视化沙盒**（2D/3D/纯文本等）。**与 AINPC 中「配置库 + REST API」属于不同层次。**

| 状态 | 说明 |
|------|------|
| **未选定** | 采用自研、复刻论文开源实现、游戏引擎、工作流编排、或其它商业/学术方案，**均未定案**。 |
| **与本文档关系** | **不阻塞** §4～§7 所描述的 AINPC **M1** 实现：人物/场景数据与关联可先落地；`simulation_meta` 为将来引擎 **回写状态** 预留，**不要求**引擎先行存在。 |
| **选定之后** | 建议另立 **《引擎选型与集成》**（或本文档附录），约定：进程边界、与 AINPC 的同步方式（轮询 / Webhook / 消息队列）、`simulation_meta` 的推荐字段约定（仍保持 JSON 自由，仅作协作规范）。 |
| **选型参考** | 常见形态对比见 **附录 A**（非约束）。 |

| 术语 | 含义 |

## 5. 非功能需求

### 5.1 性能与数据量

- 列表接口默认分页；关联查询避免 N+1（可用 JOIN 或批量查询）。
- 单场景关联 NPC 数量预期 &lt; 500；若更大需索引与分页策略。

### 5.2 安全

- 现有 API Key 等敏感字段规则不变；新增接口遵循同一鉴权策略（若未来加鉴权）。
- 用户上传资源仍走既有 `/api/upload` 能力（头像等）。

### 5.3 删除策略（已确认）

- **删除场景**：仅删除场景实体及 **场景↔NPC 关联**，**不删除** NPC 本身。
- **删除 NPC**：**禁止删除**仍存在 `scene_npc` 关联记录的 NPC。数据库层使用 **ON DELETE RESTRICT**（或等价校验）；接口返回明确错误信息，前端提示用户先在相关场景中 **解除关联** 后再删除。（**不采用**「自动解除关联并删除」。）

### 5.4 删除与危险操作前的校验（已确认）

- **凡删除类操作**（删除场景、删除 NPC 等）及 **覆盖关联**（`PUT .../npcs` 全量替换）执行前，须在 **服务端执行业务校验**（如 NPC 是否仍关联场景、资源是否存在、请求体是否合法）；**禁止**在未校验通过时执行删除或写入。
- 前端宜配合二次确认（如 `ElMessageBox`），但 **以后端校验为最终依据**。

## 6. 数据模型草案

### 6.1 新表：`scene`（建议）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| name | VARCHAR(128) | 场景名称（如地点/活动名），必填 |
| description | VARCHAR(512) 或 TEXT | 简介（环境说明、剧情提要、对话上下文提示等） |
| category | VARCHAR(32) | **分类**，与 `npc.category` **同一套取值**（如 `task` / `plot` / `custom`），默认 `custom`，索引便于筛选 |
| tags | JSON NULL | **标签**（可选）：存储为 **JSON 数组**（建议元素为字符串，如 `["室内","社交"]`）；空表示无标签；列表检索按 §12 约定解析 |
| status | TINYINT | 0 禁用 / 1 启用 |
| sort | INT | 排序 |
| created_at / updated_at | DATETIME | 与现有表一致 |

> **封面图**：v0.x **暂不实现**（已确认）；若未来需要，再走 `/api/upload` 增列即可。

### 6.2 新表：`scene_npc`（多对多）

| 字段 | 类型 | 说明 |
|------|------|------|
| scene_id | BIGINT FK → scene(id) |  |
| npc_id | BIGINT FK → npc(id) |  |
| role_note | VARCHAR(256) NULL | 可选：本场景中身份/备注 |
| PRIMARY KEY (scene_id, npc_id) | | 防重复 |

索引：`idx_npc_id` 便于从 NPC 反查场景。

### 6.3 与现有 `npc` 表关系

- 多对多由 `scene_npc` 实现；**删除 NPC** 受 **RESTRICT** 约束（见 §5.3）。
- **记忆/反思（已确认）**：在 `npc` 表增加 **`simulation_meta`（JSON，可空）**，内容为 **完全自由 JSON**（**不强制 Schema**），由调用方（外部仿真或管理端）自行约定结构；**本字段不替代**完整记忆流引擎，仅为编排与外部运行时对接预留。

```sql
-- 迁移示例（实际以 migrate 脚本为准）
ALTER TABLE npc ADD COLUMN simulation_meta JSON DEFAULT NULL COMMENT '外部仿真回写：记忆/反思等摘要';
```

## 7. API 草案（REST）

前缀与现有项目一致，例如 `/api/scene`。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/scene | 列表 + 查询参数（keyword、status、**category**、tag、page、pageSize） |
| GET | /api/scene/:id | 详情，含关联 NPC 列表（或单独子资源） |
| POST | /api/scene | 创建 |
| PUT | /api/scene/:id | 更新 |
| DELETE | /api/scene/:id | 删除（策略见 5.3） |
| PUT | /api/scene/:id/npcs | **覆盖整表**：请求体表示该场景下 **完整的** NPC 关联列表（含可选 `role_note`）；执行后 `scene_npc` 中该 `scene_id` 的旧记录 **全部替换** 为本次提交（见 §12） |

NPC 侧可增加：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/npc/:id/scenes | 某 NPC 关联的场景列表（**后续迭代**；首版可仅用场景详情反查） |

现有 **NPC 创建/更新/详情** 接口需支持字段 **`simulation_meta`**（JSON，可选，见 §6.3）。

响应格式与现有 `code/message/data` 保持一致。

## 8. 前端改动范围（Vue）

- `App.vue`：新增 Tab **「场景」**，或采用「角色 NPC」子路由/子菜单（按 UI 稿定）。
- 新增页面/组件：`SceneList.vue`、`SceneForm.vue`（或复用 Element Plus 表单模式，与 `NpcList`/`NpcForm` 一致）；场景 **分类** 复用与 NPC 一致的选项（见 `constants/npc.ts` 中 `NPC_CATEGORIES` 或抽取共享常量）。
- **首版**：`NpcForm` / `NpcList` **不要求**展示所属场景（见 §4.3）；**`simulation_meta`** 可仅在高级折叠区或调试区编辑（任意 JSON 文本），避免普通用户误改。
- **可视化**：**不与 pixel-agents 联动**；仅可借鉴其交互灵感，**有更好的 UI/引擎方案时可完全不用该项目**。

## 附录 A：常见引擎选型方向（参考，非约束）

> 本节为 **§1.4 主引擎** 的选型思路整理，**不构成功能或采购约束**；具体项目名、版本以当时社区状态为准。

### A.1 按形态划分

| 方向 | 典型形态 | 适合场景 | 与 AINPC 的关系 |
|------|----------|----------|-----------------|
| **无「空间引擎」的编排** | 后端定时任务 / 队列 + LLM API；多智能体轮流或并行调用 | 快速验证记忆、反思、对话链；**无**二维地图与移动 | 引擎侧拉取 NPC/场景/关联；状态写入 `simulation_meta` |
| **2D Web 可视化** | Canvas / WebGL（如 Phaser、PixiJS、自研）；像素风可参考 pixel-agents **思路** | 浏览器内「小镇」轻量展示、状态同步 | 同上；前端或 BFF 调 AINPC REST |
| **游戏引擎** | Unity、Godot、Unreal 等 | 需要路径寻路、碰撞、复杂动画、3D/强交互 | 游戏内嵌模块或独立进程经 HTTP/消息总线同步 AINPC |
| **学术/论文复现向** | 与 *Generative Agents* 相关的开源沙盒或复现仓库 | 贴近论文行为、可对比实验 | 常需二次开发；数据层仍可对接 AINPC |
| **多智能体应用框架** | LangGraph、AutoGen、CrewAI、Semantic Kernel Agents 等 | **对话与工具编排**、子代理分工 | **不等于**空间仿真；可与「空间引擎」组合使用（框架负责推理链，引擎负责世界状态） |
| **工作流 / 低代码** | n8n、Dify、Make 等 | 原型、人机审批、简单调度 | 适合 PoC；复杂涌现与长时记忆通常需回退到代码 |

### A.2 选型时可对照的维度（自检清单）

- **空间与时间**：是否需要二维/三维坐标、离散时间步、全局时钟？  
- **记忆与状态**：记忆流存在引擎库、向量库还是仅回写 AINPC `simulation_meta`？  
- **团队栈**：前端/游戏/后端现有能力是否匹配？  
- **运维**：单机演示 vs 长期在线服务；是否需要队列与横向扩展？  
- **授权与依赖**：引擎与依赖库许可证、云 API 成本。

### A.3 与本文档其它条款的衔接

- **pixel-agents**：仅作交互灵感，见 §10、§12；不是必选技术栈。  
- **主引擎未定**：仍可在 AINPC 内完成人物、场景、API（§4～§7）；引擎定案后通过 **《引擎选型与集成》** 固化对接方式（见 §1.4）。
