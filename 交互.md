# AINPC 交互与地图需求

> 参考：AI-Town（斯坦福小镇升级版）源码解读，a16z 主导

---

## 定稿决策

| 项 | 定稿 |
|----|------|
| **产品定位** | **游戏**，地图与交互为**核心玩法** |
| **地图渲染** | **Phaser** |
| **空地图** | tick 跳过，不执行 |
| **单人 NPC** | 仅执行反思（个人组），**不触发对话** |
| **WebSocket** | **需校验权限**，多用户**有隔离** |
| **开发顺序** | 先地图渲染 + NPC 位置展示 → 再 tick + 对话逻辑 |

## 零、存储架构：MySQL + Redis 混合

### 0.1 职责划分

| 存储 | 职责 | 数据特点 |
|------|------|----------|
| **MySQL** | 持久化、关系型、可查询分析 | 配置、NPC、记忆、对话历史、地图定义 |
| **Redis** | 实时状态、高频读写 | 位置、动作状态、对话组、tick 上下文 |

### 0.2 数据归属

| 数据 | 存储 | 说明 |
|------|------|------|
| ai_config、npc | MySQL | 已有，不动 |
| npc_memory | MySQL | 已有，记忆持久化 |
| npc_conversation、npc_message | MySQL | 已有，用户-NPC 对话历史 |
| npc_npc_conversation、npc_npc_message | MySQL | 新增，NPC 对话历史持久化 |
| game_map | MySQL | 地图配置，变更少 |
| **NPC 位置 (x, y)** | **Redis** | tick 内高频更新 |
| **NPC 动作状态 (state)** | **Redis** | moving/speaking/thinking |
| **对话组** | **Redis** | divideIntoGroups 结果，会话期间有效 |
| **当前对话消息** | **Redis**（可选） | 进行中的 NPC 对话，结束后写入 MySQL |

### 0.3 数据流与 WebSocket

```
tick 循环 → 读 Redis 位置/状态 → 分组、决策、LLM 调用
         → 写 Redis 位置/状态/对话组
         → WebSocket 推送给前端（位置、状态、新消息）
         → 对话结束/定时 → 持久化到 MySQL（npc_npc_conversation、npc_npc_message）
```

### 0.4 技术要点

| 组件 | 实现 |
|------|------|
| Redis 客户端 | Node.js: ioredis 或 node-redis |
| WebSocket | ws 或 Socket.IO，按 mapId 或 room 订阅 |
| tick 驱动 | setInterval 或 node-cron，可配置间隔（如 3–5 秒/tick） |

## 二、交互与行为

### 2.1 角色动作状态

参考 AI-Town Player 组件的三种核心状态：

| 状态 | 英文 | 说明 |
|------|------|------|
| 移动 | Moving | 在地图上行走、寻路 |
| 交谈中 | Speaking | 正在参与对话，展示说话动画 |
| 思考 | Thinking | 等待决策或处理中 |

> 相比斯坦福 AI 小镇的丰富动作与地图交互，AI-Town 更侧重“Agents 与 Agents 交谈”的框架，动作状态由后端 `playerState` 控制。

### 2.2 交互类型

| 类型 | 说明 | 优先级 |
|------|------|--------|
| 用户 ↔ NPC | 玩家与单个 NPC 对话（已实现雏形） | P0 |
| NPC ↔ NPC | NPC 之间自动对话、社交 | P1 |
| 地图交互 | 点击移动、碰撞、区域触发 | P1 |
| 多 NPC 同时在场 | 支持多人对话组、空间分组 | P2 |

### 2.3 空间与分组

AI-Town 的 `divideIntoGroups` 逻辑：根据 NPC 之间的**相对距离**分为：

- **对话组**：距离近的**多个** NPC（≥2）可进行多人对话
- **个人组**：单人 NPC **仅执行反思**，不触发对话

> **定稿**：地图上仅 1 个 NPC 时，归入个人组，只执行 reflectOnMemories，不触发 converse。

## 三、NPC 之间的聊天（多 Agent 对话）

### 3.1 整体流程（参考 AI-Town handleAgentInteraction）

```
tick 循环
  ├── 前置：若 map 无绑定 NPC（空地图）→ 跳过本 tick
  ├── divideIntoGroups：按距离分成「对话组」（≥2 人）和「个人组」（1 人）
  ├── 对话组处理：
  │   ├── 查 relationship 记忆（NPC 间关系）
  │   ├── decideWhoSpeaksNext：LLM 决定下一个发言者
  │   ├── walkAway：LLM 决定是否有人离开对话
  │   ├── startConversation：首次打招呼 + 相关记忆
  │   ├── converse：实际对话生成（身份 + 关系 + 少量记忆）
  │   └── rememberConversation：总结对话并入库
  └── 个人组处理：
      └── reflectOnMemories：重要性累计 > 阈值时做反思
```

### 3.2 核心能力需求

| 能力 | 说明 | 优先级 |
|------|------|--------|
| 发言顺序决策 | 由 LLM 根据聊天记录决定下一个说话的 NPC | P1 |
| 离开决策 | 由 LLM 判断是否有人离开当前对话 | P2 |
| 开场/寒暄 | 首次见面时打招呼，注入关系记忆 | P1 |
| 对话生成 | 多轮对话，身份 + 关系 + 最多 2 条 reflection + 2 条 conversation 记忆 | P1 |
| 对话记忆 | 每轮对话结束后总结并写入 memories | P1 |
| 反思机制 | 重要性累计达阈值时，从近期记忆提炼高层级 insight | P2 |

### 3.3 记忆与上下文控制

| 策略 | 说明 |
|------|------|
| 关系记忆 | memories 表中 type=relationship，存 NPC 间的 like/dislike 等 |
| 记忆裁剪 | 对话时仅取 2 条 reflection + 2 条 conversation，避免 token 爆炸 |
| 对话总结 | 每轮结束用 LLM 总结，写入 type=conversation |
| 反思触发 | importanceScore 总和 > 500 时触发 reflectOnMemories |

### 3.4 成本与优化（AI-Town 经验）

- **问题**：prompt 中 memory 未过滤，直接拼接超长字符串，token 消耗大
- **数据**：8 个 Agent 满速对话，约 5 美元/小时（gpt-3.5-turbo-16k）
- **优化**：对 memory 做重要性筛选、摘要压缩后再注入；严格控制注入条数（如 2–5 条）

## 四、数据库设计（MySQL + Redis）

### 4.1 MySQL：持久化数据

#### 4.1.1 地图配置（game_map）

| 表名 | 用途 | 核心字段 |
|------|------|----------|
| **game_map** | 地图配置 | id, name, width, height, tile_data(JSON), metadata |

```sql
CREATE TABLE IF NOT EXISTS game_map (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(128) NOT NULL COMMENT '地图名称',
  width INT NOT NULL COMMENT '地图宽度（格子数）',
  height INT NOT NULL COMMENT '地图高度（格子数）',
  tile_data JSON NOT NULL COMMENT '2D 数组，如 [[0,1,2],[1,0,1]]，0=可行走 1=障碍',
  metadata JSON DEFAULT NULL COMMENT '扩展：图块类型、图层等',
  status TINYINT(1) DEFAULT 1,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='游戏地图表';
```

#### 4.1.2 已有表（不动）

| 表 | 职责 |
|----|------|
| ai_config、npc | 配置、角色定义 |
| npc_memory | 记忆（conversation/reflection/relationship） |
| npc_conversation、npc_message | 用户-NPC 对话历史 |

### 4.2 Redis：实时状态

> NPC 位置、状态、对话组由 Redis 承载，tick 内高频读写；对话结束后持久化到 MySQL。

#### 4.2.1 Key 设计

| Key 模式 | 类型 | 说明 | TTL 示例 |
|----------|------|------|----------|
| `map:{mapId}:npc:{npcId}` | Hash | NPC 位置与状态 | 会话期间 |
| `map:{mapId}:npcs` | Set | 当前地图上的 NPC id 列表 | 会话期间 |
| `group:{groupId}:members` | Set | 对话组内的 NPC id | 对话期间 |
| `group:{groupId}:messages` | List | 当前对话消息（可选） | 对话期间 |
| `map:{mapId}:groups` | Set | 当前地图的活跃对话组 id | 会话期间 |

#### 4.2.2 NPC 状态 Hash 结构

```
map:{mapId}:npc:{npcId}
  x              int    格子 X
  y              int    格子 Y
  state          str    idle|moving|speaking|thinking
  groupId        str    当前对话组 id，空表示未在对话
  targetX        int    移动目标 X（可选）
  targetY        int    移动目标 Y（可选）
  updatedAt      str    时间戳
```

#### 4.2.3 对话组结构

```
group:{groupId}
  memberIds      str    JSON 数组，如 [1,2,3]
  status         str    active|ended
  createdAt      str    时间戳
```

#### 4.2.4 可选：定时持久化到 MySQL

- 对话结束时：将 npc_npc_conversation、npc_npc_message 写入 MySQL
- 定期快照：可将 Redis 中的 NPC 位置写入 MySQL 表 `npc_map_state` 做存档（非必须）

#### 4.1.3 NPC 与 NPC 对话（MySQL 持久化）

> 进行中的对话在 Redis；**结束后**写入 MySQL 做历史查询与分析。

```sql
-- NPC 与 NPC 的对话会话表（持久化用）
CREATE TABLE IF NOT EXISTS npc_npc_conversation (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  map_id BIGINT DEFAULT NULL,
  participant_ids JSON NOT NULL COMMENT '如 [1,2,3]',
  status ENUM('active','ended') DEFAULT 'ended',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  ended_at DATETIME DEFAULT NULL,
  INDEX idx_status (status),
  INDEX idx_map (map_id),
  FOREIGN KEY (map_id) REFERENCES game_map(id) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='NPC之间对话会话表（持久化）';

-- NPC 与 NPC 的对话消息表（持久化用）
CREATE TABLE IF NOT EXISTS npc_npc_message (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  conversation_id BIGINT NOT NULL,
  speaker_npc_id BIGINT NOT NULL,
  content TEXT NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_conversation_id (conversation_id),
  FOREIGN KEY (conversation_id) REFERENCES npc_npc_conversation(id) ON DELETE CASCADE,
  FOREIGN KEY (speaker_npc_id) REFERENCES npc(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='NPC之间对话消息表（持久化）';
```

### 4.3 可选：MySQL 存档表（npc_map_state）

> 若需存档/回放，可定期将 Redis 中的位置快照写入此表；非必须。

```sql
CREATE TABLE IF NOT EXISTS npc_map_state (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  npc_id BIGINT NOT NULL,
  map_id BIGINT NOT NULL,
  x INT NOT NULL DEFAULT 0,
  y INT NOT NULL DEFAULT 0,
  state VARCHAR(32) DEFAULT 'idle',
  snapshot_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '快照时间',
  INDEX idx_map_snapshot (map_id, snapshot_at),
  FOREIGN KEY (npc_id) REFERENCES npc(id) ON DELETE CASCADE,
  FOREIGN KEY (map_id) REFERENCES game_map(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='NPC地图状态快照（可选存档）';
```

### 4.4 npc 与地图绑定（npc_map_binding）

```sql
CREATE TABLE IF NOT EXISTS npc_map_binding (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  npc_id BIGINT NOT NULL,
  map_id BIGINT NOT NULL,
  init_x INT NOT NULL DEFAULT 0,
  init_y INT NOT NULL DEFAULT 0,
  UNIQUE KEY uk_npc_map (npc_id, map_id),
  FOREIGN KEY (npc_id) REFERENCES npc(id) ON DELETE CASCADE,
  FOREIGN KEY (map_id) REFERENCES game_map(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='NPC与地图绑定及初始位置';
```

### 4.5 存储汇总

| 存储 | 数据 | 用途 |
|------|------|------|
| MySQL | game_map | 地图配置 |
| MySQL | npc_map_binding | NPC 与地图绑定、初始位置 |
| MySQL | npc_npc_conversation, npc_npc_message | NPC 对话历史持久化 |
| MySQL | npc_memory 等 | 已有，不变 |
| **Redis** | **map:{mapId}:npc:{npcId}** | **NPC 实时位置、状态** |
| **Redis** | **group:{groupId}** | **对话组、进行中消息** |
| Redis | map:{mapId}:npcs, map:{mapId}:groups | 地图内 NPC 列表、对话组列表 |

## 六、实现优先级与开发顺序（已定稿）

**开发顺序**：先地图渲染 + NPC 位置展示 → 再 tick + 对话逻辑

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| 1 | 地图：2D 数组 + Phaser 渲染，NPC 位置与移动 | P1 |
| 2 | WebSocket 推送 + 鉴权 + 多用户隔离 | P1 |
| 3 | NPC-NPC：双人对话（1v1），converse + rememberConversation | P1 |
| 4 | NPC-NPC：多人对话组，divideIntoGroups + decideWhoSpeaksNext | P2 |
| 5 | 反思：reflectOnMemories，importance 累计触发 | P2 |
| 6 | 地图交互：点击移动、区域触发、更丰富动作 | P2 |

## 七、推荐方案与设计补充

### 7.1 启动与恢复（推荐）

| 项 | 推荐方案 |
|----|----------|
| **Redis 冷启动** | 场景启动时从 MySQL 读取 `game_map` + `npc_map_binding`（见 7.5），将 NPC 初始位置写入 Redis；若 binding 未配置，则从地图中心或随机可行走格子生成 |
| **Redis 重启** | 短期可接受「重启后重新初始化」；若需恢复，在场景结束时或每 N 分钟将 Redis 位置快照写入 `npc_map_state`，重启时从最新快照恢复 |

### 7.2 用户（玩家）与地图（推荐）

| 项 | 推荐方案 |
|----|----------|
| **用户是否在地图内** | **首期不落地图**：用户不写入 Redis，仅作为旁观者查看 NPC 行为；后期若需「用户在地图内移动」，再增加 `user:{userId}:map:{mapId}` 的 Redis Hash |
| **用户-NPC 对话触发** | **点击 NPC 打开对话面板**：用户在地图/列表中点选 NPC，打开对话面板，触发已有 HTTP 对话 API |
| **HTTP 与 WebSocket 分工** | **HTTP `POST /chat`**：继续用于用户-NPC 对话（面板内）；**WebSocket**：仅推送地图内 NPC 位置、状态、NPC-NPC 对话消息，前端按需订阅 |

### 7.3 部署与扩展（推荐）

| 项 | 推荐方案 |
|----|----------|
| **多实例** | 首期**单实例**即可；扩展到多实例时，使用 **Redis Pub/Sub**：tick 结果发布到 `map:{mapId}:events`，各实例 WebSocket 订阅并推送给对应连接的客户端 |
| **多地图** | 按 `mapId` 隔离 key（`map:{mapId}:*`），同一 Redis 支持多地图；场景结束时对 `map:{mapId}:*` 批量 DEL 或设置 TTL |

### 7.4 性能与限流（推荐）

| 项 | 推荐方案 |
|----|----------|
| **tick 与 LLM 延迟** | **每 tick 限流**：每 tick 最多处理 2–3 个对话组，其余顺延到下一 tick；LLM 调用**异步**执行，不阻塞 tick 循环；使用**队列**（如 Bull + Redis）管理 LLM 任务，避免并发过载 |
| **成本控制** | 配置项：`max_llm_calls_per_tick`（如 3）、`max_llm_calls_per_day`（可选）；超限时跳过本轮 LLM 调用，记录日志 |

### 7.5 数据与配置（推荐）

| 项 | 推荐方案 |
|----|----------|
| **NPC 与地图关联** | 新增 **`npc_map_binding`** 表（见 4.4）：`npc_id`、`map_id`、`init_x`、`init_y`，支持同一 NPC 出现在多地图、不同初始位置 |
| **relationship 的 related_ids** | 约定 JSON 结构：`{"target_npc_id": 2}` 或 `{"npc_id": 2}`；查询「NPC A 与 NPC B 的关系」时按 `target_npc_id` 过滤 |

### 7.6 WebSocket 鉴权与多用户隔离（已定稿）

| 项 | 方案 |
|----|------|
| **鉴权** | 连接建立时校验 token/session，未通过则拒绝或关闭连接 |
| **权限** | 用户仅能订阅其有权限访问的 mapId（如：所在游戏房间、所属地图） |
| **多用户隔离** | 按 `userId` + `mapId` 或 `roomId` 隔离：同一地图多观众时，每人仅收到该 map 的广播，不互窜；敏感数据（如有）不推送给无权限用户 |

**实现要点**：连接 URL 携带 token，服务端解析后校验；按 `map:{mapId}` 或 `room:{roomId}` 做订阅分组，仅向有权限的客户端推送。

### 7.7 补充设计：场景初始化与运维

#### 场景初始化流程

```
1. 加载 game_map（MySQL）
2. 查询 npc_map_binding WHERE map_id = ?，获取参与 NPC 及 init_x, init_y
3. 若绑定数量 = 0（空地图）→ 不启动 tick，跳过
4. 若 Redis 中无 map:{mapId}:npcs，则遍历 binding，对每个 NPC：
   - HSET map:{mapId}:npc:{npcId} x init_x y init_y state idle groupId "" updatedAt now
   - SADD map:{mapId}:npcs {npcId}
5. 启动 tick 定时任务
```

#### Redis Key TTL

| Key 模式 | TTL 建议 |
|----------|----------|
| `map:{mapId}:*` | 场景结束时主动 DEL；或设 24 小时 TTL，心跳续期 |
| `group:{groupId}` | 对话结束时 DEL；或 1 小时 TTL |

#### 错误处理

| 场景 | 处理 |
|------|------|
| LLM 调用失败 | 重试 1 次；仍失败则记录日志，该 NPC 本 tick 保持 thinking，下 tick 再试 |
| Redis 连接断开 | 重连；重连前 tick 暂停或降级为仅读 MySQL（若有快照） |
| WebSocket 断开 | 客户端重连后从 Redis 拉取当前全量状态（`map:{mapId}:npcs` 遍历获取位置） |

#### 监控指标（建议）

- tick 执行耗时（P50、P99）
- 每 tick LLM 调用次数、失败率
- Redis 内存占用、key 数量
- WebSocket 连接数、消息推送延迟
